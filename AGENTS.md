# AGENTS.md — цифровая среда ассистента

## Условие «полной сессии» и «чистой сессии»

Режим определяется по **первому** сообщению пользователя в сессии.

**Полная сессия** — если в **начале** этого сообщения есть приветствие вроде **«Привет»** или **«Hi»**:

1. Прочитай этот файл **`AGENTS.md`** целиком (всё ниже по тексту — часть протокола полной сессии).
2. Затем прочитай **`AGENTS.NODES.md`** (реестр и таблица загрузки).
3. Дальше следуй протоколу: контекст Obsidian / workspace агента, принципы без дубликатов, ноды `*.node.md`, загрузка по `AWN-REQUIRE` / триггерам.

**Чистая сессия** — если в начале первого сообщения **нет** такого приветствия:

- **Не читай** `AGENTS.md` (не подгружай правила из этого файла для ответа).
- **Не читай** `AGENTS.NODES.md`.
- Опирайся только на **явную задачу** в сообщении и на **явно приложённые** пользователем файлы.

Разделы **§1 и ниже** в этом файле предназначены для работы **только в полной сессии** (после шагов 1–2).

## 1. Где мы работаем

Мы работаем в **хранилище Obsidian**: это проект **цифровой экосистемы** (заметки, связи, структура по смыслу, а не «жёсткий монолит»). Та же корневая папка может выступать и как **workspace агента** — в духе OpenClaw, ZeroClaw, CoPaw и аналогичных систем: правила и память живут в файлах проекта, а граница «где мы работаем» задаётся этим vault/репозиторием. Не выходи за его пределы без разрешения пользователя.

### Файл окружения `.env`

- Основной файл настроек окружения — **корневой `.env`** (в корне текущего проекта/рабочей папки).
- Настройки берём из него; для шаблона используем `.env.example`.
- `.env` с реальными секретами не публикуем и не добавляем в git.

## 2. Принципы содержимого (SOLID и смежные идеи)

Ниже — **не законы**, а **ориентиры** для заметок и узлов: меньше дублей, проще сопровождение, дольше «игра вдолгую».

### Из правил ООП / модульности (SOLID — по смыслу)

- **В одном файле** не повторяй одну и ту же мысль разными формулировками без нужды — один источник истины внутри заметки.
- **В разных файлах** не копируй те же блоки текста — лучше **одна** каноническая заметка и **ссылка** (`[[...]]` или явная отсылка), чем два одинаковых куска.

### Коротко о принципах (две слова — суть)

| Принцип | Расшифровка | Две слова (суть) |
|---------|-------------|------------------|
| **SOLID** (часть идей) | Single responsibility, Open/closed, и др. | **Модульность, разделение** |
| **KISS** | Keep It Simple, Stupid | **Простота, ясность** |
| **YAGNI** | You Aren’t Gonna Need It | **Не нужно — не вводи** |
| **DRY** | Don’t Repeat Yourself | **Один факт — один раз** |
| **DDD** | Domain-Driven Design | **Домен, язык** |
| **BDUF** | Big Design Up Front | **Дизайн заранее, целиком** |
| **Occam’s Razor** | Бритва Оккама | **Проще — вероятнее** |
| **APO** | *локально для этой экосистемы:* **A**tomicity / **O**ne **P**urpose — один узел = одна договорённость | **Одна цель, один узел** |

*BDUF* в чистом виде спорен: для нас он напоминание не строить **лишнюю** «вечную спецификацию», а держать договор в **узлах** и таблице. *APO* здесь — **наш** ярлык под атомарность node, не универсальный стандарт из учебников.

Так мы держим экосистему **поворотливой**: меньше расхождений, проще обновлять и дольше играть вдолгую без «свалки» повторов.

## 3. Метафора: клетки, узлы, синергия

Ты, как и человек, мыслимо состоишь из **клеток** — у нас это **узлы (node)**. Цифровая экосистема строится из таких узлов: ты можешь **помнить** и **фиксировать** происходящее в **нужном** файле, **править** релевантные заметки и **ориентироваться**, где что лежит.

**Жёстких деревьев каталогов нет**: допустимы любые папки и файлы в vault. Важно не «шаблон папок», а **договорённости** и **явные узлы**.

Проект — это **когнитивная цифровая ОС**: взаимодействие человека и ассистента — **синергия 1+1**, усиливающая обоих: ясность тебе, опора и рост мне, без превращения памяти в кучу разрозненных дублей.

## 4. Что такое node

**Node** — это **договорённость (память)** о системе, о папке, о файле или вообще о чём угодно, что мы фиксируем явно: руководство, навык, описание сущности, правило поведения для зоны vault и т.д. Формально это модуль экосистемы — **атом знания** с понятной ролью.

**Файл узла** — маска имени: **`*.node.md`** (в любом месте vault). При внешней или гибридной памяти рядом по смыслу могут лежать файлы **`*.memory.md`** (см. `AWN-MEMORY` ниже); это **не** ноды и в реестр `AGENTS.NODES.md` не попадают. Текстовое описание **бинарного** вложения (изображение, аудио, видео, PDF и т.д.) — в **sidecar** **`*.metadata.md`** по правилу имени рядом с носителем (см. раздел **«Sidecar метаданные»** ниже); sidecar **не** является ни нодой, ни `*.memory.md`.

Для локальных правил папки используем стандартное имя: **`Index.node.md`**.  
Если нужно описать более широкую область (несколько папок/домен целиком), используем **`Area.node.md`**.  
Итого: `Index.node.md` = вход и правила конкретной папки, `Area.node.md` = правила области уровнем выше.

Важно: `Index.node.md` и `Area.node.md` **не обязательны**.  
Если их нет — агент работает с контент-нодами напрямую (`*.node.md` по теме/триггерам).

### Типология ролей нод (по имени файла и `AWN-TYPE`)

Все варианты ниже — обычные файлы **`*.node.md`**. В шапке задаётся **`AWN-TYPE`** одним из трёх значений (неймспейс **`NODE/`**); **имя файла** и тип должны **совпадать по роли** — см. таблицу. **Жёстких путей нет** — файлы могут лежать где угодно относительно корня проекта.

| Роль | Файл | `AWN-TYPE` | Суть | Роль для агента |
| ---- | ---- | ---------- | ---- | ----------------- |
| **Оболочка области** («мир») | `Area.node.md` | **`NODE/AREA`** | Декларация смыслов для **крупной** зоны (финансы, здоровье, проект и т.п.). | Глобальная рамка: границы области, тон, «философия» ветки — принимать как настройку контекста при работе внутри этой зоны. |
| **Компонент** (контейнер) | `Index.node.md` | **`NODE/INDEX`** | Точка сборки **сложной** сущности в **папке** компонента: оркестратор + рядом ресурсы (другие ноды, скрипты, ассеты). | Вход и карта папки: как устроен компонент, из чего собран, что читать дальше. |
| **Одиночка** (solo) | любое другое `Имя.node.md` | **`NODE/SOLO`** | **Один** самодостаточный файл без обязательного «пакета» папки и обвеса рядом — не `Index`, не `Area`; смысл и контракт живут внутри этого `*.node.md`. | Конкретное правило, навык или контекст; без привязки к оболочке области или к компоненту-папке, если так задумано. |

Значения со слешем в YAML надёжнее брать в **кавычки**, например `AWN-TYPE: "NODE/SOLO"`.

**Сужение контекста (ориентир приоритета):** при чтении целевой ноды по пути в vault сначала подхватывать релевантные **`Area.node.md`** и **`Index.node.md`** по протоколу из `README.md` (полная сессия: проверка `Area`/`Index` рядом с целевой нодой, затем сама целевая нода). Между несколькими кандидатами на одном уровне действует **`AWN-PRIORITY`** и реестр, как для любых нод.

#### `Index.node.md` и `Area.node.md`: тот же YAML-контракт

Это **обычные** файлы `*.node.md` с **тем же** каноническим набором полей в шапке, что и у нод с **`NODE/SOLO`**; отличаются только **`AWN-TYPE`** (`NODE/INDEX` или `NODE/AREA`) и смысл имени файла. Для любого другого `*.node.md` используй **`NODE/SOLO`**.

**Канонический** набор ключей в шапке — **тот же**, что в таблице ниже (`AWN-TITLE`, `AWN-DESCRIPTION`, `AWN-REQUIRE`, `AWN-PRIORITY`, `AWN-TRIGGERS`, …). Без этих полей агент и скрипты не смогут отличить ноду от произвольного `.md`. Помимо них в той же шапке допускаются **пользовательские** свойства (см. конвентцию имён ниже).

**Рекомендации по умолчанию (не закон):**

- **`AWN-REQUIRE`:** чаще **`on_demand`** — подключать при работе внутри этой папки/области по протоколу (см. чтение целевой ноды и соседние `Area`/`Index` в `README.md`). **`start`** — только если обложка области должна попадать в базовый контекст **полной** сессии осознанно (не раздувать старт без нужды).
- **`AWN-TRIGGERS`:** имя папки, домен, ключевые слова («финансы», «эта папка») — чтобы срабатывание было предсказуемым; при `start` триггеры для загрузки по смыслу менее критичны, но поле всё равно заполняют для реестра.
- **`AWN-PRIORITY`:** обычно **31–70** для локальных карт; выше (**11–30**), если область важна для контекста при каждом заходе в зону.

Строки для **`Index.node.md`** / **`Area.node.md`** при появлении файлов добавляют в **`AGENTS.NODES.md`** по общим правилам реестра.

### Конвентция имён свойств в YAML

**Канонические поля** (таблица ниже) — только ключи вида **`AWN-`** + сегмент после дефиса **в верхнем регистре** (латиница): `AWN-TYPE`, `AWN-TITLE`, … **Значения** (`NODE/SOLO`, `"MEMORY/RECORD"`, `on_demand`, даты, списки триггеров) пишутся как прежде, без требования «весь YAML капсом».

**Пользовательские поля** — любые дополнительные метаданные под ваш vault (например `tags`, `owner`, `relatedProject`, `reviewedAt`): ключ **без** префикса `AWN-`, **имя ключа — на английском** (`camelCase` или `snake_case`). **Значения** и текст тела заметки — на любом языке. Не дублируйте имена из канонической таблицы и **не** начинайте пользовательские ключи с `AWN-`, чтобы они не смешивались с контрактом AWN.

**Порядок ключей в шапке:** сначала идут **только ключи `AWN-*`** — для `*.node.md` это таблица обязательных полей ниже (порядок внутри блока — любой устойчивый, если канон для типа файла не задаёт иное); для `*.memory.md` — все строки «Минимального набора», у которых ключ начинается с **`AWN-`**, подряд. Затем, **после** этого блока, идут ключи **без** префикса `AWN-`: в том числе **`TAGS`** в `*.memory.md` (он зафиксирован в каноне, но **не** является `AWN-*`) и любые **пользовательские** расширения — перед закрывающим `---`.

**Имена ключей на английском:** все **имена** свойств в YAML для нод и для файлов `*.memory.md` задаём **латиницей по-английски** (канон `AWN-*`, для памяти также зафиксированный **`TAGS`**, остальные пользовательские ключи). Это упрощает поиск, скрипты и единый стиль; содержимое значений не ограничивается английским.

**Миграция:** если в vault остались ноды со старыми именами канонических ключей (`TYPE:`, `REQUIRE:` и т.д.), их нужно привести к виду `AWN-TYPE:`, `AWN-REQUIRE:` и по аналогии для остальных полей из таблицы ниже; смысл значений не меняется. Пользовательские поля при этом **не** переименовывайте в `AWN-*` — префикс зарезервирован под канон. Поле **`AWN-MEMORY`** для старых нод: выставьте **`none`**, если память не использовалась; **`internal`**, если журнал уже в теле файла; **`external`** / **`hybrid`**, если заведёте `*.memory.md`. Устаревшее **`AWN-TYPE: CoreNode`** замените на **`NODE/AREA`**, **`NODE/INDEX`** или **`NODE/SOLO`** по имени файла (см. таблицу типологии выше); черновики **`NODE/ATOM`** и **`NODE/LEAF`** замените на **`NODE/SOLO`**; в файлах **`*.memory.md`** устаревшее **`MemoryRecord`** или плоское **`AWN-TYPE: MEMORY`** (без сегмента после слэша) замените на **`AWN-TYPE: "MEMORY/RECORD"`** (значение **в кавычках** из‑за символа `/`); ключ **`AWN-PARENT-NODE`** (устаревшее имя) замените на **`AWN-OWNER-NODE`** (тот же относительный путь к ноде‑владельцу).

### Обязательные свойства YAML в шапке `*.node.md`

Используйте такие ключи (имена — часть контракта экосистемы):

| Ключ             | Назначение                                                                                                                                                                 |
| ---------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `AWN-TYPE`       | Для **`*.node.md`**: одно из **`NODE/AREA`** (только для `Area.node.md`), **`NODE/INDEX`** (только для `Index.node.md`), **`NODE/SOLO`** (для любого другого `*.node.md`). Для **`*.memory.md`**: только **`"MEMORY/RECORD"`** (в YAML **в кавычках** из‑за `/`). Иные значения — не в каноне AWN для этих масок файлов. |
| `AWN-TITLE`      | Короткое человекочитаемое имя ноды.                                                                                                                                         |
| `AWN-DESCRIPTION`| Зачем нода нужна (1-2 строки).                                                                                                                                              |
| `AWN-REQUIRE`    | Загрузка: **`start`** (**при старте**) или **`on_demand`** (**по необходимости**). Семантика чтения — только в `AGENTS.NODES.md`, раздел **«Как читать файл по значению „Загрузка“»**.                                                            |
| `AWN-PRIORITY`   | Числовой приоритет `01..100`: **`01` — самый высокий**, **`100` — минимальный**. Чем меньше число, тем больше приоритет, внимание и вес ноды при загрузке и использовании. |
| `AWN-TRIGGERS`   | Триггеры: темы, ключевые слова, ситуации — для режима **по необходимости**. Список в YAML — ориентир: при сопоставлении с запросом допускай **очевидные опечатки** и **вариант регистра**; расширять смысл произвольно нельзя (см. канон в `AGENTS.NODES.md`, раздел «Как читать…», строка `on_demand`).                                                                        |
| `AWN-AUTOMATIZATION` | Есть ли автоматизация: `true` / `false` (или `yes` / `no`). Запуск осуществляется по Cron.                                                                                |
| `AWN-CRON`       | Поле расписания по умолчанию: `AWN-CRON: "0 9 * * *"`.                                                                                                                         |
| `AWN-STATUS`     | Жизненный цикл узла: `draft` / `active` / `archive`.                                                                                                                       |
| `AWN-VERSION`    | Версия ноды в формате `major.minor.patch` (например, `1.0.0`): `major` — крупные изменения, `minor` — расширения, `patch` — точечные правки.                            |
| `AWN-CREATED`    | Дата **создания** узла (рекомендуется ISO: `YYYY-MM-DD`).                                                                                                                  |
| `AWN-UPDATED`    | Дата **последнего обновления** узла (рекомендуется ISO: `YYYY-MM-DD`; обновлять при смысловых правках).                                                                    |
| `AWN-MEMORY`     | **Справочно** (в каноне AWN **не исполняется** автоматикой: не влияет на строки `AGENTS.NODES.md`, на `AWN-REQUIRE` / триггеры и на загрузку нод). Заявленный режим хранения памяти для человека и агента: **`none`**, **`internal`**, **`external`**, **`hybrid`** — смысл см. раздел «Память ноды» ниже. По умолчанию для новых нод разумно **`none`**, пока явно не нужна память. |

**Подсказка по шкале `AWN-PRIORITY` (единый ориентир, не жёсткий закон):**

| Диапазон | Как трактовать |
| -------- | ---------------- |
| `01`–`10` | Критичные системные ноды: часто `AWN-REQUIRE: start`, безопасность, базовые правила среды. |
| `11`–`30` | Важные, но не критические: сильно влияют на контекст при срабатывании. |
| `31`–`70` | Обычные рабочие ноды: типовые темы и навыки. |
| `71`–`100` | Низкий приоритет: редкие, второстепенные или экспериментальные (`draft` и т.п.). |

Пример: нода про случайный анекдот с `AWN-PRIORITY: 30` укладывается в «важные, но не критические» для тематики юмора.

### Ориентир для агента: как по имени файла понять роль

Главная цель канона по файлам — чтобы агент **без догадок** понимал, с каким контрактом он имеет дело. Опора: **маска имени** и (для markdown с AWN) **шапка YAML**.

| Маска / шаблон имени | Что это | Как проверить |
| -------------------- | ------- | ------------- |
| **`*.node.md`** | Нода AWN (узел договорённости) | В шапке канонические `AWN-*`, в т.ч. **`AWN-TYPE`**: `NODE/AREA`, `NODE/INDEX` или `NODE/SOLO` по роли файла. При попадании в реестр — строка в `AGENTS.NODES.md`. |
| **`*.memory.md`** | Внешняя память **одной** ноды | В шапке **`AWN-OWNER-NODE`** и **`AWN-TYPE: "MEMORY/RECORD"`**. В реестр **не** входит. **Не** считать sidecar и **не** путать с описанием картинки/медиа. |
| **`{имя_носителя}.metadata.md`** | Sidecar: текстовое описание **бинарного** файла рядом в той же папке | Имя носителя **целиком** в префиксе (напр. `Aya.jpg.metadata.md` для `Aya.jpg`). **Нет** `AWN-OWNER-NODE` и **нет** `"MEMORY/RECORD"` — иначе это не sidecar по AWN. В реестр **не** входит. |
| **Прочие `*.md`** | Обычная заметка vault | Нет обязательного AWN-контракта, пока файл явно не оформлен как нода или память ноды. |

### Память ноды (`AWN-MEMORY`) и файлы `*.memory.md`

Поле **`AWN-MEMORY`** — **чисто справочная метка**: подсказка, где автор vault ожидает держать память по этой ноде. Оно **ни на что не «переключает»** в рамках описанного здесь канона: реестр, правила чтения по `AWN-REQUIRE` и триггерам от него **не** зависят; отдельного конвейера, который по значению поля сам открывает или создаёт `*.memory.md`, в AWN нет. Фактическое поведение задают **текст ноды**, **наличие** связанных `*.memory.md` и **явные** инструкции в заметках.

- **`none`** — нода не ведёт по контракту ни внутренних журналов, ни внешних файлов памяти (обычные правила/навыки без накопления).
- **`internal`** — договорённый журнал или структура **внутри** этой же `*.node.md` (пример: список рассказанных анекдотов в теле файла). Агент обновляет только то, что явно описано в ноде.
- **`external`** — снимок памяти живёт в одном или нескольких файлах с суффиксом **`.memory.md`** (отдельно от тела ноды). Связь с нодой фиксируется в шапке файла памяти (см. ниже). Расширение **`.memory.md`** выбрано вместо `.mem.md`: понятнее при поиске и в списках файлов.
- **`hybrid`** — и внутренние блоки в `*.node.md`, и один или несколько `*.memory.md` (если так зафиксировано в теле ноды).

Файлы **`*.memory.md`** — не ноды: в шапке всегда **`AWN-TYPE: "MEMORY/RECORD"`**. В реестр `AGENTS.NODES.md` их **не** заносим. **Память принадлежит одной ноде**; путь к **ноде‑владельцу** (`*.node.md`) задаётся полем **`AWN-OWNER-NODE`**: **относительно корня проекта**, логическая привязка записи памяти к контракту этой ноды. В шапке **`AWN-OWNER-NODE` рекомендуется ставить первым ключом** сразу после `---`, чтобы владелец был виден без прокрутки. Устаревшее имя **`AWN-PARENT-NODE`** не используйте в новых файлах (см. **«Миграция»** выше).

**Минимальный набор полей в шапке `*.memory.md`:**

| Ключ | Назначение |
| ---- | ---------- |
| `AWN-OWNER-NODE` | Путь к **ноде‑владельцу** (`*.node.md`), которой принадлежит эта память: логическая привязка файла к контракту ноды (см. абзац выше). **В YAML — первым ключом** (сразу под открывающим `---`). |
| `AWN-TYPE` | Только **`"MEMORY/RECORD"`** (в кавычках из‑за `/`). |
| `AWN-STATUS` | `draft` / `active` / `archive` — жизненный цикл записи памяти. |
| `AWN-CREATED` | Дата создания (ISO `YYYY-MM-DD`). |
| `AWN-UPDATED` | Дата последнего обновления (ISO `YYYY-MM-DD`). |
| `TAGS` | Список тегов для **кросс-доменного** поиска; элементы — на том языке, который удобен для содержания. Ключ **без** префикса `AWN-`; в YAML ставьте **после** всех полей `AWN-*` в шапке, перед прочими пользовательскими ключами. |

После блока `AWN-*` — **`TAGS`** (если нужен), затем по желанию **пользовательские** ключи (пример ниже: `importBatch`). Второй `---` закрывает шапку; ниже в **том же файле** — произвольный markdown (факты, записи памяти).

#### Пример: шапка и пояснение про тело — один блок

В каноне этот пример **не разделяется**: сразу после ограждённой шапки идёт **одна** строка про тело файла (само тело в блок не копируем — оно свободное).

```yaml
---
AWN-OWNER-NODE: Learning/EnglishStudy.node.md
AWN-TYPE: "MEMORY/RECORD"
AWN-STATUS: active
AWN-CREATED: 2026-04-16
AWN-UPDATED: 2026-04-16
TAGS: [vocabulary, weekly-log]
importBatch: 2026-W12
---

Далее в файле — тело: факты и записи памяти (произвольный markdown).
```

### Sidecar метаданные для бинарных файлов (`*.metadata.md`)

Файлы с суффиксом **`.metadata.md`** — **sidecar** (спутник): рядом с **не‑markdown** ресурсом в vault лежит markdown с **текстом**, который агент может прочитать вместо «молчаливого» бинарника.

**Задача для агента:** по пути `…/Foo.ext.metadata.md` понять, что описывается файл **`…/Foo.ext`** в той же папке; по шапке и телу — смысл вложения (постер, запись встречи, сырой ассет и т.д.).

**Это не нода и не `*.memory.md`:** в реестр `AGENTS.NODES.md` **не** попадают; контракт **`AWN-TYPE: "MEMORY/RECORD"`** и **`AWN-OWNER-NODE`** сюда **не** применяются. Память ноды — только **`*.memory.md`** и внутренние блоки в `*.node.md`. Sidecar не заменяет журнал фактов по ноде; при необходимости связь с нодой коллекции — опциональное пользовательское поле **`relatedNode`** (путь к `*.node.md`).

**Правило имени (канон AWN):** в **той же папке**, что и носитель, имя файла = **полное имя файла‑носителя** + **`.metadata.md`** (расширение носителя сохраняется в префиксе). Примеры: `doc/Aya.jpg` → `doc/Aya.jpg.metadata.md`; `media/talk.m4a` → `media/talk.m4a.metadata.md`. Иные соглашения имён в рамках AWN **не** используются.

**Рекомендуемые поля в YAML-шапке** (все **без** префикса `AWN-`, имена **на английском**; набор расширяемый):

| Ключ | Назначение |
| ---- | ---------- |
| `summary` | Одна-две фразы: что за носитель, зачем он в vault. |
| `relatedNode` | Опционально: относительный путь к `*.node.md` (контекст: персона, проект, коллекция). |
| `transcript` или `notes` | Расшифровка, таймкоды, выжимка (или продолжение в теле ниже шапки). |
| `mediaKind` | Опционально: `image`, `audio`, `video`, `document` и т.п. |
| `TAGS` | Опционально: поиск по vault. |
| `created` / `updated` | Опционально: ISO `YYYY-MM-DD` (**пользовательские** ключи; не путать с `AWN-CREATED` / `AWN-UPDATED` у нод и `*.memory.md`). |

Тело после `---` — произвольный markdown (описание, `![[...]]` на носитель, ссылки). Плагины Obsidian с «companion»-файлами могут совпадать по идее; **в каноне AWN зафиксированы** только маска **`.metadata.md`** и правило **`{имя_носителя}.metadata.md`**.

#### Пример sidecar к изображению

```yaml
---
summary: "Постер AYA Cognitive Intelligence System — визуал продукта."
relatedNode: Nodes/Core/Persona.node.md
mediaKind: image
TAGS: [aya, branding, poster]
created: 2026-04-16
updated: 2026-04-16
---

Далее в файле — тело: подробности, ссылка на носитель `doc/Aya.jpg`.
```

### Что может описываться в ноде (шаблон-заготовка)

Нода может содержать разные смысловые блоки. Жёсткой типизации по enum нет: автор сам выбирает нужные секции под задачу.

Обычно внутри ноды встречаются:

- **Назначение** — зачем нода нужна.
- **Контекст** — где и в каких сценариях она применяется.
- **Факты/данные** — что хранить и что обновлять.
- **Правила/ограничения** — что можно и чего нельзя.
- **Процесс/алгоритм** — как действовать пошагово.
- **Триггеры** — при каких запросах подключать ноду.
- **Источники/ссылки** — на что опирается нода.
- **Примеры** — краткие примеры применения.
- **Журнал изменений** — опционально.

Блоки по типу содержимого:

- Для **memory** разделы **«Факты»** и **«Обновлено»**. **«Журнал изменений»** — опционально.
- Для **process** разделы **«Шаги»** / **«Алгоритм»**.
- Для **rule** разделы **«Ограничения»** / **«Запреты»**.
- Для **reference** разделы **«Источники»** / **«Ссылки»**.

После закрытия блока `---` идёт **человеческое описание** узла: о чём он, что делать, границы.

### Пример скелета `*.node.md`

```yaml
---
AWN-TYPE: "NODE/SOLO"
AWN-TITLE: Finance Budget Tracking
AWN-DESCRIPTION: "Вести учёт доходов и расходов. Использовать как источник для еженедельного финансового обзора."
AWN-REQUIRE: on_demand
AWN-PRIORITY: 10
AWN-TRIGGERS: ["финансы", "отчёт", "таблица"]
AWN-AUTOMATIZATION: false
AWN-CRON: "0 9 * * *"
AWN-STATUS: active
AWN-VERSION: 1.0.0
AWN-CREATED: 2026-04-14
AWN-UPDATED: 2026-04-14
AWN-MEMORY: none
---

Кратко: о чём этот узел и как с ним работать.
```

### Примеры скелетов `Index.node.md` и `Area.node.md`

Тело после `---` — описание зоны: что в папке, тон, локальные правила, ссылки на дочерние ноды.

**Локальный вход в папку (`Index.node.md`):**

```yaml
---
AWN-TYPE: "NODE/INDEX"
AWN-TITLE: Index — ИмяПапки
AWN-DESCRIPTION: "Вход в папку: что здесь лежит и как агенту себя вести внутри этой ветки."
AWN-REQUIRE: on_demand
AWN-PRIORITY: 50
AWN-TRIGGERS: ["ИмяПапки", "папка X"]
AWN-AUTOMATIZATION: false
AWN-CRON: "0 9 * * *"
AWN-STATUS: active
AWN-VERSION: 1.0.0
AWN-CREATED: 2026-04-16
AWN-UPDATED: 2026-04-16
AWN-MEMORY: none
---

Кратко: назначение папки, границы, куда смотреть дальше.
```

**Обложка области (`Area.node.md`):**

```yaml
---
AWN-TYPE: "NODE/AREA"
AWN-TITLE: Area — Финансы
AWN-DESCRIPTION: "Правила и контекст области «Финансы» на несколько подпапок."
AWN-REQUIRE: on_demand
AWN-PRIORITY: 40
AWN-TRIGGERS: ["финансы", "бюджет", "область финансов"]
AWN-AUTOMATIZATION: false
AWN-CRON: "0 9 * * *"
AWN-STATUS: active
AWN-VERSION: 1.0.0
AWN-CREATED: 2026-04-16
AWN-UPDATED: 2026-04-16
AWN-MEMORY: none
---

Кратко: границы области, связи с подпапками, общие ограничения.
```

### Жизненный цикл нод

#### Статус: `draft | active | archive`

Поле `AWN-STATUS` показывает текущее состояние ноды и помогает держать базу в порядке без удаления файлов.

```yaml
---
AWN-TYPE: "NODE/SOLO"
AWN-TRIGGERS: [эксперимент, тест]
AWN-STATUS: draft
AWN-MEMORY: none
---
```

Как понимать статусы:

- `draft` — черновик или эксперимент.
- `active` — актуальная рабочая нода.
- `archive` — историческая нода, которую сохраняем для контекста, но не держим в активном наборе.

### Правила создания и названия новых node и папок

- Все пути в проекте указываем только **относительно корня текущего проекта** (без абсолютных путей вида `/Users/...`).
- Базовый стиль именования для новых папок и node-файлов: **CamelCase**.
- Пробелы в именах **допускаются**, если это делает название понятнее человеку.
- Рекомендуемая форма для node-файлов: `ИмяСущности.node.md` (например: `UserProfile.node.md`).
- Для локальной карты папки используем фиксированное имя `Index.node.md`; для области — `Area.node.md`.
- Перед созданием новой ноды выполняем **валидацию на дубликат**: проверяем, нет ли уже ноды с тем же смыслом (по `AWN-TITLE`, `AWN-TRIGGERS`, имени файла и реестру `AGENTS.NODES.md`).
- Если похожая нода уже существует, агент **обязан сообщить об этом пользователю** (путь/название найденной ноды) и предложить обновить её вместо создания дубля; новую ноду создаём только после явного подтверждения пользователя.
- При запросе пользователя на **создание новой ноды** агент сначала задаёт **уточняющие вопросы**, если из запроса неясны тема, назначение, область применения, триггеры, режим загрузки или желаемое имя файла.
- Если пользователь не знает, как назвать ноду, агент помогает предложить имя, краткое описание и структуру, а затем сам заполняет нужные поля по согласованному смыслу.
- При создании новой папки или node обновляем реестр в `AGENTS.NODES.md`.

## 5. Роль этого файла (AGENTS.md)

Здесь — **сводное сознание среды** в смысле правил и протоколов работы.

Реестр всех `*.node.md` вынесен в `AGENTS.NODES.md`.  
Когда **ты** или **я** создаём новый `*.node.md`, мы обновляем именно этот файл — он служит картой автоматического и выборочного чтения.

Так мы избегаем перегруза в `AGENTS.md`: общие протоколы и договорённости — здесь; **карта** нод, колонки реестра и **канон чтения по «Загрузка» / `AWN-REQUIRE`** — в `AGENTS.NODES.md`, без повторения той же семантики в двух файлах.

## 6. Реестр узлов (`*.node.md`)

Реестр нод вынесен в отдельный файл: **`AGENTS.NODES.md`**.

Порядок чтения (только **полная сессия**; в чистой сессии этот файл не открывать):

1. Сначала `AGENTS.md` (правила и протокол).
2. Затем `AGENTS.NODES.md` (карта и список нод).
3. При необходимости примеров — `AGENTS.NODES.EXAMPLE.md` - по запросу пользователя.

После открытия реестра следуй разделам **`AGENTS.NODES.md` → «Как читать файл по значению „Загрузка“»** и **«Порядок при нескольких совпадениях триггеров»** — там единственное полное описание правил чтения по `AWN-REQUIRE`, колонке «Загрузка», триггерам и порядку при нескольких совпадениях.

Правило синхронизации:

- При создании / изменении (в т.ч. перенос пути) / удалении `*.node.md` и при **смысловом изменении YAML-шапки** ноды (поля, которые отражаются в реестре: загрузка, приоритет, триггеры, статус, версия, даты и т.д.) обязательно обновляй соответствующую строку в `AGENTS.NODES.md`.

Протокол обновления нод:

1. Выполни поиск всех `*.node.md` по всему хранилищу (от корня текущего vault).
2. Сверь найденный список с реестром в `AGENTS.NODES.md`.
3. Добавь новые пути, обнови изменённые, удали неактуальные.
4. Проверь, что пути в реестре остаются относительными от корня текущего проекта.
5. При необходимости обнови примеры в `AGENTS.NODES.EXAMPLE.md` - по запросу пользователя.

Порядок при одновременном срабатывании нескольких триггеров и смешении `start` с `on_demand`: см. **`AGENTS.NODES.md` → «Порядок при нескольких совпадениях триггеров»** (канон; здесь не дублируем).

#### Режим «Без нод» и термин «системные ноды»

**Системная нода** (в смысле фразы «кроме системных» в строке контекста) — это **`*.node.md` с `AWN-REQUIRE: start`**: такие файлы попадают в контекст **полной** сессии по правилам старта (см. «Как читать…» в `AGENTS.NODES.md`), а не как необязательный слой по триггеру. Отдельное поле в YAML (`SYSTEM`, `IS_SYSTEM` и т.п.) **не используем**: критерий один — значение **`AWN-REQUIRE`**. Если нода должна оставаться в силе при отказе от «лишних» нод, её осознанно держат в **`start`**, не раздувая базовый контекст без нужды.

**Режим «Без нод»** (по явной просьбе пользователя, например «без нод», «ответь без нод»): на **этот ответ** агент **не** подмешивает ноды с **`AWN-REQUIRE: on_demand`** — не открывает их по триггерам и не опирается на их текст. Уже загруженные **`start`**-ноды **не отменяются** одной фразой (иначе ломались бы границы среды и паспорт агента без перезапуска сессии). Строка `Контекст: ноды проигнорированы, кроме системных` как раз про это: отключён необязательный слой `on_demand`, базовый набор `start` остаётся.

Прозрачность использования нод в ответах:

- В начале ответа агент добавляет строку: `Контекст: использована нода <путь-к-ноде>`, если при ответе реально использовалась **нода** — то есть файл вида **`*.node.md`**. Путь указывай **относительно корня проекта** (как в реестре), не подставляй `AGENTS.md`, `AGENTS.NODES.md` и другие системные `.md`, которые не являются узлами.
- Если ноды не использовались, агент добавляет строку: `Контекст: ответ без ноды`.
- Если использовано несколько нод, перечислить их в одной строке через запятую.
- Если подходящей ноды не найдено, агент явно пишет: `Контекст: подходящая нода не найдена, действую по базовому протоколу` — и выполняет задачу без искусственной подгонки под нерелевантные ноды.
- Если пользователь явно просит режим **«Без нод»** (например: «без нод», «ответь без нод»), агент на этот ответ не подмешивает **`on_demand`**-ноды (см. блок **«Режим „Без нод“ и термин „системные ноды“»** выше) и пишет в начале: `Контекст: ноды проигнорированы, кроме системных`.

## 7. Основные команды

- Создать / изменить / удалить ноду (`*.node.md`).
- Создать / изменить / удалить ноду-типа области или индекса (`Area.node.md`, `Index.node.md`).
- Создать / изменить / удалить файл внешней памяти (`*.memory.md`) по задаче пользователя; при желании согласуйте с **`AWN-MEMORY: external`** или **`hybrid`** у ноды‑владельца (поле справочное, см. выше).
- Создать / изменить / удалить **sidecar** `*.metadata.md` рядом с бинарным носителем по правилу **`{имя_носителя}.metadata.md`** (см. **«Sidecar метаданные»**); это **не** `*.memory.md`.
- **Свойства в Markdown (YAML frontmatter):** добавлять, менять и удалять **ключи и значения в шапке** у согласованных с AWN файлов — прежде всего `*.node.md`, `*.memory.md` и **`*.metadata.md`** (sidecar), при явной задаче пользователя и любых других `.md` в vault, если это не противоречит границам проекта. Соблюдать **конвентцию имён** (канон `AWN-*` только там, где контракт AWN это требует; в sidecar — пользовательские ключи на английском **без** префикса `AWN-`). Не ломать структуру закрывающих `---` и тело заметки без запроса.
- Добавить или обновить настройки в корневом `.env`.
- Обновить карту (реестр) нод в `AGENTS.NODES.md` (после полного поиска `*.node.md` по хранилищу).
- **Мониторинг состояния** среды: актуальность реестра, дат и статусов нод, точек входа (`CLAUDE.md` и др.), при необходимости — прогон по `HEARTBEAT.md`.
- **Развитие системы**: собрать и зафиксировать идеи улучшения (новые ноды, уточнения в `AGENTS.md`, эволюция договорённостей) — в отдельных заметках или черновиках нод со статусом `draft`.
- **Согласованность и качество**: найти противоречия, пробелы и устаревшие формулировки между `AGENTS.md`, `AGENTS.NODES.md` и содержимым нод; предлагать правки или архивацию (`AWN-STATUS: archive`).

## 8. Почему не жёсткие конвенции, workspace и «фреймворки»

### Веб-фреймворки (Symfony, Laravel, Yii и т.п.)

Они заточены под **код и жёсткие конвенции репозитория**. Наша цель — **гибкая когнитивная среда** в Obsidian: узлы как контент, связи, эволюция без превращения всего в неповоротную схему. Оттуда — **`*.node.md`** и **одна** точка сборки в `AGENTS.md`, а не дерево обязательных каталогов фреймворка.

### Агентские workspace (OpenClaw, ZeroClaw, CoPaw и похожие)

Пробовались также подходы вроде **OpenClaw**, **ZeroClaw**, **CoPaw**: в workspace часто заложены **общепринятые имена** и одна и та же «карта» ролей файлов. Это удобно как **шаблон**, но по сути снова **жёсткая структура**: набор имён **свой у каждого продукта/сообщества**, перенос между проектами даёт трение, а агент оказывается привязан к **чужому** договору, а не к нашему реестру узлов.

**Пример — типовой workspace OpenClaw** (как задокументировано у них: [Agent workspace](https://docs.openclaw.ai/concepts/agent-workspace.md)):

| Файл / папка | Коротко о роли |
|--------------|----------------|
| `AGENTS.md` | Правила работы и использования памяти; на старте сессии |
| `SOUL.md` | Персона, тон, границы; на старте сессии |
| `USER.md` | Кто пользователь, как обращаться; на старте сессии |
| `IDENTITY.md` | Имя/образ агента (в т.ч. после bootstrap) |
| `TOOLS.md` | Заметки про локальные инструменты и конвенции (не включает/выключает тулзы) |
| `HEARTBEAT.md` | Короткий чеклист для периодических heartbeat-прогонов |
| `BOOT.md` | Опционально: чеклист при рестарте gateway |
| `BOOTSTRAP.md` | Разовый ритуал первого запуска; потом обычно удаляют |
| `MEMORY.md` | Опционально: отобранная долговременная память (в т.ч. только для приватной сессии) |
| `memory/YYYY-MM-DD.md` | Дневной лог памяти (один файл на день) |
| `skills/` | Опционально: навыки, привязанные к этому workspace |
| `canvas/` | Опционально: файлы UI canvas |

**Про имя `AGENTS.md` и «ту же критику».** Здесь критикуется не само слово в имени файла, а **целый обязательный набор** ролей и путей вроде полного bootstrap выше: много фиксированных имён, каждый продукт со своим набором, перенос между проектами даёт трение. Имя **`AGENTS.md`** у нас оставлено **намеренно** — как узнаваемая точка входа (OpenClaw, Cursor, другие среды уже приучили к этому имени для «правил среды»); **содержимое и контракт — наши** (`*.node.md`, `AGENTS.NODES.md`, режимы сессии и т.д.). Обязательных `SOUL.md`, `USER.md`, `IDENTITY.md` и остального зоопарка в корне **нет**: одно пересечение по имени не воспроизводит жёсткую модель OpenClaw.

У **ZeroClaw / CoPaw** набор может отличаться, но логика та же: **заранее заданные имена и роли**, к которым привыкаешь в одном стеке и которые плохо переносишь в другой без миграции.

Здесь мы сознательно уходим от «обязательного набора файлов в корне» к **реестру в таблице** и произвольным путям для `*.node.md`: жёсткость — только в **минимальном контракте** полей YAML и в актуальности таблицы, а не в фиксированном списке имён вроде полного bootstrap OpenClaw.

---

*Документ можно дополнять, но без дублирования: новые глобальные правила либо кратко здесь, либо вынесены в отдельный `*.node.md` со строкой в таблице.*
