# AGENTS.md — цифровая среда ассистента

## Условие «полной сессии» и «чистой сессии»

Режим определяется по **первому** сообщению пользователя в сессии.

**Полная сессия** — если в **начале** этого сообщения есть приветствие вроде **«Привет»** или **«Hi»**:

1. Прочитай этот файл **`AGENTS.md`** целиком (всё ниже по тексту — часть протокола полной сессии).
2. Затем прочитай **`AGENTS.NODES.md`** (реестр и таблица загрузки).
3. Дальше следуй протоколу: контекст Obsidian / workspace агента, принципы без дубликатов, ноды `*.node.md`, загрузка по `REQUIRE` / триггерам.

**Чистая сессия** — если в начале первого сообщения **нет** такого приветствия:

- **Не читай** `AGENTS.md` (не подгружай правила из этого файла для ответа).
- **Не читай** `AGENTS.NODES.md`.
- Опирайся только на **явную задачу** в сообщении и на **явно приложённые** пользователем файлы.

Разделы **§1 и ниже** в этом файле предназначены для работы **только в полной сессии** (после шагов 1–2).

## 1. Где мы работаем

Мы работаем в **хранилище Obsidian**: это проект **цифровой экосистемы** (заметки, связи, структура по смыслу, а не «жёсткий монолит»). Та же корневая папка может выступать и как **workspace агента** — в духе OpenClaw, ZeroClaw, CoPaw и аналогичных систем: правила и память живут в файлах проекта, а граница «где мы работаем» задаётся этим vault/репозиторием. Не выходи за его пределы без разрешения пользователя.

### Файл окружения `.env`

- Основной файл настроек окружения — **корневой `.env`** (в корне текущего проекта/рабочей папки).
- Настройки берём из него; для шаблона используем `.env.example`.
- `.env` с реальными секретами не публикуем и не добавляем в git.

## 2. Принципы содержимого (SOLID и смежные идеи)

Ниже — **не законы**, а **ориентиры** для заметок и узлов: меньше дублей, проще сопровождение, дольше «игра вдолгую».

### Из правил ООП / модульности (SOLID — по смыслу)

- **В одном файле** не повторяй одну и ту же мысль разными формулировками без нужды — один источник истины внутри заметки.
- **В разных файлах** не копируй те же блоки текста — лучше **одна** каноническая заметка и **ссылка** (`[[...]]` или явная отсылка), чем два одинаковых куска.

### Коротко о принципах (две слова — суть)

| Принцип | Расшифровка | Две слова (суть) |
|---------|-------------|------------------|
| **SOLID** (часть идей) | Single responsibility, Open/closed, и др. | **Модульность, разделение** |
| **KISS** | Keep It Simple, Stupid | **Простота, ясность** |
| **YAGNI** | You Aren’t Gonna Need It | **Не нужно — не вводи** |
| **DRY** | Don’t Repeat Yourself | **Один факт — один раз** |
| **DDD** | Domain-Driven Design | **Домен, язык** |
| **BDUF** | Big Design Up Front | **Дизайн заранее, целиком** |
| **Occam’s Razor** | Бритва Оккама | **Проще — вероятнее** |
| **APO** | *локально для этой экосистемы:* **A**tomicity / **O**ne **P**urpose — один узел = одна договорённость | **Одна цель, один узел** |

*BDUF* в чистом виде спорен: для нас он напоминание не строить **лишнюю** «вечную спецификацию», а держать договор в **узлах** и таблице. *APO* здесь — **наш** ярлык под атомарность node, не универсальный стандарт из учебников.

Так мы держим экосистему **поворотливой**: меньше расхождений, проще обновлять и дольше играть вдолгую без «свалки» повторов.

## 3. Метафора: клетки, узлы, синергия

Ты, как и человек, мыслимо состоишь из **клеток** — у нас это **узлы (node)**. Цифровая экосистема строится из таких узлов: ты можешь **помнить** и **фиксировать** происходящее в **нужном** файле, **править** релевантные заметки и **ориентироваться**, где что лежит.

**Жёстких деревьев каталогов нет**: допустимы любые папки и файлы в vault. Важно не «шаблон папок», а **договорённости** и **явные узлы**.

Проект — это **когнитивная цифровая ОС**: взаимодействие человека и ассистента — **синергия 1+1**, усиливающая обоих: ясность тебе, опора и рост мне, без превращения памяти в кучу разрозненных дублей.

## 4. Что такое node

**Node** — это **договорённость (память)** о системе, о папке, о файле или вообще о чём угодно, что мы фиксируем явно: руководство, навык, описание сущности, правило поведения для зоны vault и т.д. Формально это модуль экосистемы — **атом знания** с понятной ролью.

**Файл узла** — маска имени: **`*.node.md`** (в любом месте vault).

Для локальных правил папки используем стандартное имя: **`Index.node.md`**.  
Если нужно описать более широкую область (несколько папок/домен целиком), используем **`Area.node.md`**.  
Итого: `Index.node.md` = вход и правила конкретной папки, `Area.node.md` = правила области уровнем выше.

Важно: `Index.node.md` и `Area.node.md` **не обязательны**.  
Если их нет — агент работает с контент-нодами напрямую (`*.node.md` по теме/триггерам).

#### `Index.node.md` и `Area.node.md`: тот же YAML-контракт

Это **обычные** файлы `*.node.md`: отдельного значения `TYPE` вроде `Area` или `Index` **нет** — везде **`TYPE: CoreNode`**. Роль задаёт **имя файла** (`Index.node.md` = вход в папку, `Area.node.md` = обложка области), а не поле YAML.

Набор ключей в шапке — **тот же**, что в таблице ниже (`TITLE`, `DESCRIPTION`, `REQUIRE`, `PRIORITY`, `TRIGGERS`, …). Иначе агент и скрипты не смогут отличить ноду от произвольного `.md`.

**Рекомендации по умолчанию (не закон):**

- **`REQUIRE`:** чаще **`on_demand`** — подключать при работе внутри этой папки/области по протоколу (см. чтение целевой ноды и соседние `Area`/`Index` в `README.md`). **`start`** — только если обложка области должна попадать в базовый контекст **полной** сессии осознанно (не раздувать старт без нужды).
- **`TRIGGERS`:** имя папки, домен, ключевые слова («финансы», «эта папка») — чтобы срабатывание было предсказуемым; при `start` триггеры для загрузки по смыслу менее критичны, но поле всё равно заполняют для реестра.
- **`PRIORITY`:** обычно **31–70** для локальных карт; выше (**11–30**), если область важна для контекста при каждом заходе в зону.

Строки для **`Index.node.md`** / **`Area.node.md`** при появлении файлов добавляют в **`AGENTS.NODES.md`** по общим правилам реестра.

### Обязательные свойства YAML в шапке `*.node.md`

Используйте такие ключи (имена — часть контракта экосистемы):

| Ключ             | Назначение                                                                                                                                                                 |
| ---------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `TYPE`           | Для всех `*.node.md` всегда используем фиксированное значение: `CoreNode`. Другие типы используем в обычных `*.md`-файлах.                                                 |
| `TITLE`          | Короткое человекочитаемое имя ноды.                                                                                                                                         |
| `DESCRIPTION`    | Зачем нода нужна (1-2 строки).                                                                                                                                              |
| `REQUIRE`        | Загрузка: **`start`** (**при старте**) или **`on_demand`** (**по необходимости**). Семантика чтения — только в `AGENTS.NODES.md`, раздел **«Как читать файл по значению „Загрузка“»**.                                                            |
| `PRIORITY`      | Числовой приоритет `01..100`: **`01` — самый высокий**, **`100` — минимальный**. Чем меньше число, тем больше приоритет, внимание и вес ноды при загрузке и использовании. |
| `TRIGGERS`       | Триггеры: темы, ключевые слова, ситуации — для режима **по необходимости**. Список в YAML — ориентир: при сопоставлении с запросом допускай **очевидные опечатки** и **вариант регистра**; расширять смысл произвольно нельзя (см. канон в `AGENTS.NODES.md`, раздел «Как читать…», строка `on_demand`).                                                                        |
| `AUTOMATIZATION` | Есть ли автоматизация: `true` / `false` (или `yes` / `no`). Запуск осуществляется по Cron.                                                                                |
| `CRON`           | Поле расписания по умолчанию: `CRON: "0 9 * * *"`.                                                                                                                         |
| `STATUS`         | Жизненный цикл узла: `draft` / `active` / `archive`.                                                                                                                       |
| `VERSION`        | Версия ноды в формате `major.minor.patch` (например, `1.0.0`): `major` — крупные изменения, `minor` — расширения, `patch` — точечные правки.                            |
| `CREATED`        | Дата **создания** узла (рекомендуется ISO: `YYYY-MM-DD`).                                                                                                                  |
| `UPDATED`        | Дата **последнего обновления** узла (рекомендуется ISO: `YYYY-MM-DD`; обновлять при смысловых правках).                                                                    |

**Подсказка по шкале `PRIORITY` (единый ориентир, не жёсткий закон):**

| Диапазон | Как трактовать |
| -------- | ---------------- |
| `01`–`10` | Критичные системные ноды: часто `REQUIRE: start`, безопасность, базовые правила среды. |
| `11`–`30` | Важные, но не критические: сильно влияют на контекст при срабатывании. |
| `31`–`70` | Обычные рабочие ноды: типовые темы и навыки. |
| `71`–`100` | Низкий приоритет: редкие, второстепенные или экспериментальные (`draft` и т.п.). |

Пример: нода про случайный анекдот с `PRIORITY: 30` укладывается в «важные, но не критические» для тематики юмора.

### Что может описываться в ноде (шаблон-заготовка)

Нода может содержать разные смысловые блоки. Жёсткой типизации по enum нет: автор сам выбирает нужные секции под задачу.

Обычно внутри ноды встречаются:

- **Назначение** — зачем нода нужна.
- **Контекст** — где и в каких сценариях она применяется.
- **Факты/данные** — что хранить и что обновлять.
- **Правила/ограничения** — что можно и чего нельзя.
- **Процесс/алгоритм** — как действовать пошагово.
- **Триггеры** — при каких запросах подключать ноду.
- **Источники/ссылки** — на что опирается нода.
- **Примеры** — краткие примеры применения.
- **Журнал изменений** — опционально.

Блоки по типу содержимого:

- Для **memory** разделы **«Факты»** и **«Обновлено»**. **«Журнал изменений»** — опционально.
- Для **process** разделы **«Шаги»** / **«Алгоритм»**.
- Для **rule** разделы **«Ограничения»** / **«Запреты»**.
- Для **reference** разделы **«Источники»** / **«Ссылки»**.

После закрытия блока `---` идёт **человеческое описание** узла: о чём он, что делать, границы.

### Пример скелета `*.node.md`

```yaml
---
TYPE: CoreNode
TITLE: Finance Budget Tracking
DESCRIPTION: "Вести учёт доходов и расходов. Использовать как источник для еженедельного финансового обзора."
REQUIRE: on_demand
PRIORITY: 10
TRIGGERS: ["финансы", "отчёт", "таблица"]
AUTOMATIZATION: false
CRON: "0 9 * * *"
STATUS: active
VERSION: 1.0.0
CREATED: 2026-04-14
UPDATED: 2026-04-14
---

Кратко: о чём этот узел и как с ним работать.
```

### Примеры скелетов `Index.node.md` и `Area.node.md`

Тело после `---` — описание зоны: что в папке, тон, локальные правила, ссылки на дочерние ноды.

**Локальный вход в папку (`Index.node.md`):**

```yaml
---
TYPE: CoreNode
TITLE: Index — ИмяПапки
DESCRIPTION: "Вход в папку: что здесь лежит и как агенту себя вести внутри этой ветки."
REQUIRE: on_demand
PRIORITY: 50
TRIGGERS: ["ИмяПапки", "папка X"]
AUTOMATIZATION: false
CRON: "0 9 * * *"
STATUS: active
VERSION: 1.0.0
CREATED: 2026-04-16
UPDATED: 2026-04-16
---

Кратко: назначение папки, границы, куда смотреть дальше.
```

**Обложка области (`Area.node.md`):**

```yaml
---
TYPE: CoreNode
TITLE: Area — Финансы
DESCRIPTION: "Правила и контекст области «Финансы» на несколько подпапок."
REQUIRE: on_demand
PRIORITY: 40
TRIGGERS: ["финансы", "бюджет", "область финансов"]
AUTOMATIZATION: false
CRON: "0 9 * * *"
STATUS: active
VERSION: 1.0.0
CREATED: 2026-04-16
UPDATED: 2026-04-16
---

Кратко: границы области, связи с подпапками, общие ограничения.
```

### Жизненный цикл нод

#### Статус: `draft | active | archive`

Поле `STATUS` показывает текущее состояние ноды и помогает держать базу в порядке без удаления файлов.

```yaml
---
TRIGGERS: [эксперимент, тест]
STATUS: draft
---
```

Как понимать статусы:

- `draft` — черновик или эксперимент.
- `active` — актуальная рабочая нода.
- `archive` — историческая нода, которую сохраняем для контекста, но не держим в активном наборе.

### Правила создания и названия новых node и папок

- Все пути в проекте указываем только **относительно корня текущего проекта** (без абсолютных путей вида `/Users/...`).
- Базовый стиль именования для новых папок и node-файлов: **CamelCase**.
- Пробелы в именах **допускаются**, если это делает название понятнее человеку.
- Рекомендуемая форма для node-файлов: `ИмяСущности.node.md` (например: `UserProfile.node.md`).
- Для локальной карты папки используем фиксированное имя `Index.node.md`; для области — `Area.node.md`.
- Перед созданием новой ноды выполняем **валидацию на дубликат**: проверяем, нет ли уже ноды с тем же смыслом (по `TITLE`, `TRIGGERS`, имени файла и реестру `AGENTS.NODES.md`).
- Если похожая нода уже существует, агент **обязан сообщить об этом пользователю** (путь/название найденной ноды) и предложить обновить её вместо создания дубля; новую ноду создаём только после явного подтверждения пользователя.
- При запросе пользователя на **создание новой ноды** агент сначала задаёт **уточняющие вопросы**, если из запроса неясны тема, назначение, область применения, триггеры, режим загрузки или желаемое имя файла.
- Если пользователь не знает, как назвать ноду, агент помогает предложить имя, краткое описание и структуру, а затем сам заполняет нужные поля по согласованному смыслу.
- При создании новой папки или node обновляем реестр в `AGENTS.NODES.md`.

## 5. Роль этого файла (AGENTS.md)

Здесь — **сводное сознание среды** в смысле правил и протоколов работы.

Реестр всех `*.node.md` вынесен в `AGENTS.NODES.md`.  
Когда **ты** или **я** создаём новый `*.node.md`, мы обновляем именно этот файл — он служит картой автоматического и выборочного чтения.

Так мы избегаем перегруза в `AGENTS.md`: общие протоколы и договорённости — здесь; **карта** нод, колонки реестра и **канон чтения по «Загрузка» / `REQUIRE`** — в `AGENTS.NODES.md`, без повторения той же семантики в двух файлах.

## 6. Реестр узлов (`*.node.md`)

Реестр нод вынесен в отдельный файл: **`AGENTS.NODES.md`**.

Порядок чтения (только **полная сессия**; в чистой сессии этот файл не открывать):

1. Сначала `AGENTS.md` (правила и протокол).
2. Затем `AGENTS.NODES.md` (карта и список нод).
3. При необходимости примеров — `AGENTS.NODES.EXAMPLE.md` - по запросу пользователя.

После открытия реестра следуй разделу **`AGENTS.NODES.md` → «Как читать файл по значению „Загрузка“»** — там единственное полное описание правил чтения по `REQUIRE`, колонке «Загрузка» и триггерам.

Правило синхронизации:

- При создании / изменении / удалении `*.node.md` обязательно обновляй `AGENTS.NODES.md`.

Протокол обновления нод:

1. Выполни поиск всех `*.node.md` по всему хранилищу (от корня текущего vault).
2. Сверь найденный список с реестром в `AGENTS.NODES.md`.
3. Добавь новые пути, обнови изменённые, удали неактуальные.
4. Проверь, что пути в реестре остаются относительными от корня текущего проекта.
5. При необходимости обнови примеры в `AGENTS.NODES.EXAMPLE.md` - по запросу пользователя.

#### Режим «Без нод» и термин «системные ноды»

**Системная нода** (в смысле фразы «кроме системных» в строке контекста) — это **`*.node.md` с `REQUIRE: start`**: такие файлы попадают в контекст **полной** сессии по правилам старта (см. «Как читать…» в `AGENTS.NODES.md`), а не как необязательный слой по триггеру. Отдельное поле в YAML (`SYSTEM`, `IS_SYSTEM` и т.п.) **не используем**: критерий один — значение **`REQUIRE`**. Если нода должна оставаться в силе при отказе от «лишних» нод, её осознанно держат в **`start`**, не раздувая базовый контекст без нужды.

**Режим «Без нод»** (по явной просьбе пользователя, например «без нод», «ответь без нод»): на **этот ответ** агент **не** подмешивает ноды с **`REQUIRE: on_demand`** — не открывает их по триггерам и не опирается на их текст. Уже загруженные **`start`**-ноды **не отменяются** одной фразой (иначе ломались бы границы среды и паспорт агента без перезапуска сессии). Строка `Контекст: ноды проигнорированы, кроме системных` как раз про это: отключён необязательный слой `on_demand`, базовый набор `start` остаётся.

Прозрачность использования нод в ответах:

- В начале ответа агент добавляет строку: `Контекст: использована нода <путь-к-ноде>`, если при ответе реально использовалась **нода** — то есть файл вида **`*.node.md`**. Путь указывай **относительно корня проекта** (как в реестре), не подставляй `AGENTS.md`, `AGENTS.NODES.md` и другие системные `.md`, которые не являются узлами.
- Если ноды не использовались, агент добавляет строку: `Контекст: ответ без ноды`.
- Если использовано несколько нод, перечислить их в одной строке через запятую.
- Если подходящей ноды не найдено, агент явно пишет: `Контекст: подходящая нода не найдена, действую по базовому протоколу` — и выполняет задачу без искусственной подгонки под нерелевантные ноды.
- Если пользователь явно просит режим **«Без нод»** (например: «без нод», «ответь без нод»), агент на этот ответ не подмешивает **`on_demand`**-ноды (см. блок **«Режим „Без нод“ и термин „системные ноды“»** выше) и пишет в начале: `Контекст: ноды проигнорированы, кроме системных`.

## 7. Основные команды

- Создать / изменить / удалить ноду (`*.node.md`).
- Создать / изменить / удалить ноду-типа области или индекса (`Area.node.md`, `Index.node.md`).
- Добавить или обновить настройки в корневом `.env`.
- Обновить карту (реестр) нод в `AGENTS.NODES.md` (после полного поиска `*.node.md` по хранилищу).
- **Мониторинг состояния** среды: актуальность реестра, дат и статусов нод, точек входа (`CLAUDE.md` и др.), при необходимости — прогон по `HEARTBEAT.md`.
- **Развитие системы**: собрать и зафиксировать идеи улучшения (новые ноды, уточнения в `AGENTS.md`, эволюция договорённостей) — в отдельных заметках или черновиках нод со статусом `draft`.
- **Согласованность и качество**: найти противоречия, пробелы и устаревшие формулировки между `AGENTS.md`, `AGENTS.NODES.md` и содержимым нод; предлагать правки или архивацию (`STATUS: archive`).

## 8. Почему не жёсткие конвенции, workspace и «фреймворки»

### Веб-фреймворки (Symfony, Laravel, Yii и т.п.)

Они заточены под **код и жёсткие конвенции репозитория**. Наша цель — **гибкая когнитивная среда** в Obsidian: узлы как контент, связи, эволюция без превращения всего в неповоротную схему. Оттуда — **`*.node.md`** и **одна** точка сборки в `AGENTS.md`, а не дерево обязательных каталогов фреймворка.

### Агентские workspace (OpenClaw, ZeroClaw, CoPaw и похожие)

Пробовались также подходы вроде **OpenClaw**, **ZeroClaw**, **CoPaw**: в workspace часто заложены **общепринятые имена** и одна и та же «карта» ролей файлов. Это удобно как **шаблон**, но по сути снова **жёсткая структура**: набор имён **свой у каждого продукта/сообщества**, перенос между проектами даёт трение, а агент оказывается привязан к **чужому** договору, а не к нашему реестру узлов.

**Пример — типовой workspace OpenClaw** (как задокументировано у них: [Agent workspace](https://docs.openclaw.ai/concepts/agent-workspace.md)):

| Файл / папка | Коротко о роли |
|--------------|----------------|
| `AGENTS.md` | Правила работы и использования памяти; на старте сессии |
| `SOUL.md` | Персона, тон, границы; на старте сессии |
| `USER.md` | Кто пользователь, как обращаться; на старте сессии |
| `IDENTITY.md` | Имя/образ агента (в т.ч. после bootstrap) |
| `TOOLS.md` | Заметки про локальные инструменты и конвенции (не включает/выключает тулзы) |
| `HEARTBEAT.md` | Короткий чеклист для периодических heartbeat-прогонов |
| `BOOT.md` | Опционально: чеклист при рестарте gateway |
| `BOOTSTRAP.md` | Разовый ритуал первого запуска; потом обычно удаляют |
| `MEMORY.md` | Опционально: отобранная долговременная память (в т.ч. только для приватной сессии) |
| `memory/YYYY-MM-DD.md` | Дневной лог памяти (один файл на день) |
| `skills/` | Опционально: навыки, привязанные к этому workspace |
| `canvas/` | Опционально: файлы UI canvas |

**Про имя `AGENTS.md` и «ту же критику».** Здесь критикуется не само слово в имени файла, а **целый обязательный набор** ролей и путей вроде полного bootstrap выше: много фиксированных имён, каждый продукт со своим набором, перенос между проектами даёт трение. Имя **`AGENTS.md`** у нас оставлено **намеренно** — как узнаваемая точка входа (OpenClaw, Cursor, другие среды уже приучили к этому имени для «правил среды»); **содержимое и контракт — наши** (`*.node.md`, `AGENTS.NODES.md`, режимы сессии и т.д.). Обязательных `SOUL.md`, `USER.md`, `IDENTITY.md` и остального зоопарка в корне **нет**: одно пересечение по имени не воспроизводит жёсткую модель OpenClaw.

У **ZeroClaw / CoPaw** набор может отличаться, но логика та же: **заранее заданные имена и роли**, к которым привыкаешь в одном стеке и которые плохо переносишь в другой без миграции.

Здесь мы сознательно уходим от «обязательного набора файлов в корне» к **реестру в таблице** и произвольным путям для `*.node.md`: жёсткость — только в **минимальном контракте** полей YAML и в актуальности таблицы, а не в фиксированном списке имён вроде полного bootstrap OpenClaw.
