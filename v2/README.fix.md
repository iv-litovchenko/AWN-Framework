# AWN-Framework v2

## Что это

**AWN-Framework** (Agent-Workspace-Node) — это протокол организации цифрового хранилища, в котором человек и LLM-агент работают с одними и теми же файлами.

Основная идея: vault — это не просто набор заметок, а **когнитивная ОС**: структурированная среда, где знания, правила поведения агента, данные и память живут в одном месте и понятны как человеку, так и любому агенту.

Три ключевых элемента системы:

- **Ноды (`*.node.md`)** — атомы знания и договорённостей. Каждая нода описывает одну конкретную вещь: правило, навык, домен, схему. Агент читает их по триггерам или при старте.
- **Реестр (`AGENTS.NODES.md`)** — карта всех нод: что загружать при старте, что по запросу, какие приоритеты.
- **Домены (`Domains/`)** — пространства знаний пользователя (английский язык, финансы, проекты и т.д.). Каждый домен — папка со своим `Index.node.md`-дашбордом, заметками, файлами и памятью.

Основной просмотрщик для человека — **Obsidian**. Агент работает с теми же файлами через любую совместимую среду:

- **IDE-агенты:** Cursor, Antigravity и аналоги
- **LLM-агенты и фреймворки:** Claude (Cowork), Codex, Hermes Agent, Agent Zero, OpenClaw, QwenClaw и другие
- **API:** любой LLM с доступом к файловой системе vault

Vault не привязан к конкретному инструменту — любой агент, прочитавший `AGENTS.md`, понимает среду и может работать с хранилищем.

## Главный принцип

**Не где это лежит, а как найти.**

База знаний строится не вокруг жёсткой иерархии папок, а вокруг договорённостей: ноды (`*.node.md`), реестр (`AGENTS.NODES.md`), конвенции именования. Любая структура папок — вторична.

## Цель

База знаний должна быть **переносимой от агента к агенту**: любой LLM-агент, прочитав `AGENTS.md` и `AGENTS.NODES.md`, понимает где что лежит, как себя вести и как работать с доменами пользователя. Человек читает те же файлы в Obsidian.

### 5. Синхронизация Vault между устройствами

Как синхронизировать vault между MacBook, сервером и телефоном? Варианты: Яндекс.Диск, Google Drive, iCloud, Obsidian Sync, Syncthing, Git. Что работает надёжно с бинарными файлами в `Assets/`?

### 6. Расшаривание и доступ по HTTPS

Когда vault лежит на сервере — как дать ссылку на просмотр или скачивание документа по HTTPS? Варианты: Obsidian Publish, nginx + static, self-hosted Quartz/MkDocs.

### 7. Obsidian-скилл для агента

Будет ли помогать Obsidian-скилл агенту, чтобы искать и управлять заметками в Obsidian через API или плагин (Dataview, Local REST API)? Будет ли работать поиск "по смыслу", а не только glob, grep, read.

### 8. Index.md — оглавление всех файлов

Нужен ли глобальный `Index.md` в корне vault — список всех файлов с описанием (по аналогии с Andrej Karpathy)? Как его поддерживать актуальным — вручную или автоматически? Где это уместно создавать еще?

### 9. Access.md — управление доступом (RBAC / ACL)

Нужен ли файл `Access.node.md` с политиками доступа к заметкам: `public`, `private`, `allow: [список]`? Как агент должен учитывать эти ограничения при ответах и расшаривании?

### 10. Data/ — структурированные данные и схемы

Нужна ли отдельная папка `Data/` (или `Base/`) для хранения структурированных данных — схем, записей типа «БД», которые агент ведёт автоматически (задачи, контакты, события и т.п.)? Открытые вопросы: чем это отличается от `Domains/`, как просматривать и редактировать такие записи в Obsidian (Dataview, Obsidian Bases, CSV), когда оправдано выделять отдельную папку, а когда достаточно добавить домен в `Domains/`?

### 11. Area.node.md vs Index.node.md — нужны ли оба типа?

Оба типа живут на уровне папки, имеют одинаковые YAML-поля и по смыслу описывают «вход в зону». Различие сейчас — концептуальное: `Area` — контекст для агента (широкая зона), `Index` — дашборд для пользователя (конкретная папка с Dataview-запросами). Вопрос: достаточно ли одного типа (`NODE/INDEX`) и позволить содержимому самому определять роль файла? Или различие `NODE/AREA` / `NODE/INDEX` несёт ценность и должно остаться?

Варианты решения:

- **A — оставить как есть** (`NODE/AREA` + `NODE/INDEX`). Различие концептуально полезно: `Area` задаёт контекст/тон зоны для агента, `Index` — навигационный дашборд с Dataview для пользователя. Минус: два типа с одинаковыми полями, граница размыта.
- **B — оставить только `Index.node.md`** (`NODE/INDEX`). Одно имя, один тип — содержимое само показывает, широкая это зона или конкретный дашборд. Меньше концепций, проще объяснять новому агенту.
- **C — переименовать для ясности**: `Zone.node.md` (широкая область, `NODE/ZONE`) и `Index.node.md` (конкретный дашборд, `NODE/INDEX`). Смысл сохраняется, имена говорят сами за себя.

### 12. Repositories/ — внешние репозитории и зависимости vault

Vault знает о **знаниях** пользователя через домены, но ничего не знает о его **кодовых проектах**. Идея: добавить папку `Repositories/` и файл `awn.composer.json` для описания внешних репозиториев, с которыми работает пользователь локально.

**Структура:**

- `awn.composer.json` — машиночитаемый манифест всех репозиториев: `id`, локальный `path`, `git`-адрес, `stack`, `status`. Единственный источник истины для определений.
- `Repositories/Index.node.md` — человекочитаемый вход: таблица репозиториев, команды для агента.
- Опционально: отдельная нода на каждый крупный репозиторий с заметками, архитектурой, TODO.

**Связь с доменами (гибридный подход):**

- В `Index.node.md` домена добавляется поле `repositories: ["id"]` — только ссылка на ID, без дублирования деталей.
- Агент при работе в домене видит `repositories`, идёт в `awn.composer.json`, получает путь и стек.
- Принцип: домен объявляет **зависимость** (`require`), `awn.composer.json` объявляет **определение** — как `require` + `composer.json` в PHP.

**Пример `awn.composer.json`:**

```json
{
  "vault": "MyVault",
  "repositories": {
    "project-alpha": {
      "path": "~/Projects/alpha",
      "git": "https://github.com/user/alpha.git",
      "stack": "Next.js / Payload",
      "status": "active"
    },
    "project-beta": {
      "path": "~/Projects/beta",
      "git": "https://github.com/user/beta.git",
      "stack": "PHP 8.1 / WordPress",
      "status": "active"
    }
  }
}
```

4. Как быть с репозиториями без домена — просто в `awn.composer.json` без ноды?
