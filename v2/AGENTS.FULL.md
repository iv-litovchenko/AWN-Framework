# AGENTS.FULL.md

Полный протокол AWN-Framework v2. Читать **только** после срабатывания **полной сессии** — условия в `./AGENTS.md`, раздел «Bootstrap (обязательно)».

> **Где мы находимся:** текущая папка — это **Vault** (хранилище Obsidian). Он же является **workspace агента и пользователя**: все ноды, правила, память и контент живут внутри этой папки. Не выходи за её пределы без явного разрешения пользователя.

> **Роль этого файла:** здесь §1–§11 — ноды, принципы, команды, прозрачность. Точка входа и режим сессии — в `AGENTS.md`. Карта нод — в `AGENTS.NODES.md`. Не дублировать одно в другое.

---

## §1. Как это реально работает

LLM не имеет персистентной памяти между сессиями. Поэтому:

- **start-ноды** — читаются один раз при старте полной сессии и остаются в контексте как фиксированная константа до конца сессии.
- **on_demand-ноды** — читаются один раз за сессию при срабатывании триггера, затем остаются в истории до конца сессии.
- Нет механизма «загрузил один раз и держу в стороне». Всё что читается — оседает в контексте.

## §2. Ноды (`*.node.md`)

Нода — файл с договорённостью: правило, профиль, навык, контекст.

### Формат шапки (YAML)

Канонические поля используют префикс `AWN-` — это отделяет системный контракт от пользовательских полей. Пользовательские поля (теги, заметки, метаданные проекта) пишутся без `AWN-`, в camelCase на английском.

**Порядок ключей:** сначала все `AWN-*` поля, затем пользовательские.

```yaml
---
AWN-TYPE: "NODE/SOLO"        # NODE/SOLO | NODE/INDEX | NODE/AREA
AWN-TITLE: Короткое имя
AWN-DESC: "Зачем эта нода — 1-2 строки"
AWN-LOAD: start | on_demand
AWN-PRIORITY: 01..100        # меньше = важнее
AWN-TRIGGERS: [слово, другое слово]
AWN-STATUS: active | disabled | draft | archive
AWN-CATEGORY: general          # slug категории; general — по умолчанию; system — системные ноды агента
AWN-MEMORY: none | internal | external | hybrid
AWN-AUTOMATIZATION: false    # true если нода запускается по расписанию
AWN-CRON: "0 9 * * *"        # расписание (только если AWN-AUTOMATIZATION: true)
AWN-VERSION: 1.0.0           # major — крупные изменения, minor — расширения, patch — правки
AWN-CREATED: YYYY-MM-DD
AWN-UPDATED: YYYY-MM-DD
# пользовательские поля — после AWN-блока:
tags: [тег1, тег2]
owner: имя
---
```

Минимум: `AWN-TYPE`, `AWN-TITLE`, `AWN-DESC`, `AWN-LOAD`, `AWN-PRIORITY`, `AWN-STATUS`. Триггеры нужны только для `on_demand`.

### Типы нод

| `AWN-TYPE` | Файл | Роль |
|------------|------|------|
| `NODE/SOLO` | `Имя.node.md` | Самодостаточная нода — правило, навык, контекст; живёт без обязательного «пакета» папки |
| `NODE/INDEX` | `Index.node.md` | Точка сборки компонента: оркестратор папки — рядом могут лежать другие ноды, скрипты, ресурсы и любые файлы компонента; Index описывает из чего состоит компонент и что читать дальше |
| `NODE/AREA` | `Area.node.md` | Оболочка крупной области — задаёт тон, философию и границы для нескольких вложенных папок/нод |

`Index.node.md` и `Area.node.md` **не обязательны** — если их нет, агент работает с нодами напрямую по триггерам.

**Рекомендации по умолчанию для INDEX и AREA:**
- `AWN-LOAD`: чаще `on_demand` — подключать при работе внутри папки/области; `start` только если нужен в базовом контексте каждой сессии
- `AWN-PRIORITY`: обычно 31–70 для локальных карт; 11–30 если область критична при каждом заходе
- `AWN-TRIGGERS`: имя папки, домен, ключевые слова («workflow», «эта папка»)

### Ориентир: как по имени файла понять роль

| Маска имени | Что это | Как проверить |
|-------------|---------|---------------|
| `*.node.md` | Нода AWN | `AWN-TYPE`: `NODE/SOLO`, `NODE/INDEX` или `NODE/AREA` |
| `*.memory.md` | Запись памяти ноды | `AWN-TYPE: "RECORD/MEMORY"` + `AWN-OWNER` |
| `{имя_файла}.metadata.md` | Sidecar бинарного носителя | `AWN-TYPE: "RECORD/METADATA"`; имя носителя целиком в префиксе |
| `Notes/*.md` | Заметка домена | `AWN-TYPE: "RECORD/NOTE"` |
| Прочие `*.md` | Обычная заметка vault | Нет AWN-контракта |

### Входной блок ноды

Сразу после закрывающего `---` шапки YAML — **сильная рекомендация** (не обязательное поле канона, но агент добавляет его осознанно при создании/обновлении нод):

```markdown
# Название ноды

> [!info] О файле
> Нода (*.node.md, NODE/SOLO): зачем нода, что в ней, триггеры. Не путать с ...
```

Это помогает агенту и человеку сразу понять роль файла без чтения всего содержимого. Актуализировать при смысловых правках.

### Шаблон-заготовка: что может быть внутри ноды

Жёсткой типизации нет — автор выбирает нужные секции:

- **Назначение** — зачем нода нужна
- **Контекст** — где и в каких сценариях применяется
- **Факты/данные** — что хранить и обновлять
- **Правила/ограничения** — что можно и чего нельзя
- **Процесс/алгоритм** — как действовать пошагово
- **Триггеры** — при каких запросах подключать
- **Источники/ссылки** — на что опирается нода
- **Примеры** — краткие примеры применения
- **Журнал изменений** — опционально

По типу содержимого: `memory` → Факты + Обновлено; `process` → Шаги/Алгоритм; `rule` → Ограничения/Запреты; `reference` → Источники/Ссылки.

### Шаблоны скелетов

Готовые заготовки для каждого типа ноды — в папке [Templates/](Templates/):

- [Solo.template.md](Templates/Nodes/Solo.template.md) — одиночная нода (`NODE/SOLO`)
- [Index.template.md](Templates/Nodes/Index.template.md) — оркестратор папки (`NODE/INDEX`)
- [Area.template.md](Templates/Nodes/Area.template.md) — оболочка области (`NODE/AREA`)
- [Memory.template.md](Templates/Records/Memory.template.md) — файл внешней памяти (`*.memory.md`)
- [Metadata.template.md](Templates/Records/Metadata.template.md) — sidecar бинарного файла (`*.metadata.md`)
- [Note.template.md](Templates/Records/Note.template.md) — заметка домена (`Notes/`)

### Жизненный цикл (`AWN-STATUS`)

Статус — главный рубильник загрузки. **Агент читает ноду только если `AWN-STATUS: active`** — независимо от `AWN-LOAD` и триггеров.

| Статус | Влияние на загрузку | Когда использовать |
|--------|--------------------|--------------------|
| `active` | ✅ Загружается по правилам `AWN-LOAD` | Нода готова и работает |
| `disabled` | 🚫 Не загружается | Нода готова, но выключена — заглушка, временно не нужна |
| `draft` | 🚫 Не загружается | Нода в процессе написания — ещё не готова к работе |
| `archive` | 🚫 Не загружается | Нода устарела — файл сохраняется для истории |

**Примеры:**
- Ставишь `Assistant.node.md` в `disabled` → ассистент стартует без идентичности
- Ставишь `User.node.md` в `draft` → нода пишется, ассистент её не грузит
- Папка-заглушка пока пустая → `disabled`, включишь когда наполнится
- Нода устарела но жалко удалять → `archive`, остаётся в файлах, не грузится

### Приоритет (ориентир)

| Диапазон | Смысл |
|----------|-------|
| 01–10 | Критичные: идентичность, безопасность |
| 11–30 | Важные: сильно влияют на контекст |
| 31–70 | Рабочие: типовые темы и навыки |
| 71–100 | Второстепенные: редкие, черновики |

### Память ноды (`AWN-MEMORY`)

Поле декларирует где хранится накопленная память по этой ноде:

| Значение | Смысл |
|----------|-------|
| `none` | Нода не ведёт память — просто правило или навык |
| `internal` | Журнал/факты хранятся внутри самого `*.node.md` |
| `external` | Память в отдельном файле `*.memory.md` рядом |
| `hybrid` | И внутри ноды, и в `*.memory.md` |

Поле `AWN-MEMORY` — **справочная метка**, не исполняется автоматикой: не влияет на загрузку нод и реестр. Фактическое поведение задают текст ноды и явные инструкции в ней. По умолчанию для новых нод — `none`.

Файлы `*.memory.md` — не ноды, в реестр не попадают. Полный контракт (поля, примеры) — в `Assistant.Ai/Nodes/MemoryFiles.node.md`.

### Sidecar-метаданные (`*.metadata.md`)

Текстовое описание бинарного файла (картинка, аудио, PDF) — чтобы агент понимал его смысл без открытия. Имя: `{полное_имя_файла}.metadata.md` в той же папке.

Не нода и не `*.memory.md` — в реестр не попадает. Полный контракт (поля, примеры) — в `Assistant.Ai/Nodes/MetadataFiles.node.md`.

## §3. Принципы содержимого

Ориентиры для нод и заметок — не законы, но помогают не превратить vault в свалку.

### Три главных правила

**1. Не дублируй внутри одного файла.**
Одна мысль — один раз. Не повторяй одно и то же разными словами в разных местах одной ноды.

**2. Не дублируй между файлами — давай ссылку.**
Если факт уже есть в другой ноде — сошлись на неё, не копируй текст. Два одинаковых куска в разных файлах расходятся со временем и создают противоречия.

**3. Не вводи иерархию папок там, где её нет в реальности или она избыточна.**
Папка оправдана, если внутри несколько связанных файлов с общим контекстом. Одиночная нода не требует своей папки — она живёт там, где логично.

### Таблица принципов

| Принцип | Суть |
|---------|------|
| **DRY** | Один факт — один раз. Не дублировать текст между файлами — лучше ссылка на источник |
| **DDD** | Domain-Driven Design — домен и язык. Каждая область vault говорит на своём языке, ноды отражают реальные понятия домена |
| **KISS** | Нода должна быть понятна с первого прочтения |
| **YAGNI** | Не создавать ноду «на будущее» — только когда реально нужна |
| **SRP** (из SOLID) | Одна нода — одна договорённость. Смешивать несвязанные темы в одном файле — плохо |
| **APO** | Atomicity / One Purpose — атом знания с понятной ролью |
| **BDUF** | Не строить вечную спецификацию заранее — держать договор в нодах, обновлять по мере роста |
| **Occam's Razor** | Проще — вероятнее. Не плодить ноды и поля без реальной нужды |

## §4. Правила загрузки on_demand

Канон — в `AGENTS.NODES.md`, раздел «Как читать». Кратко: читать ноду только при **явном** совпадении запроса с триггером; при нескольких совпадениях — по `AWN-PRIORITY`.

**Режим «Без нод»:** если пользователь явно пишет «без нод» — пропустить все `on_demand`-ноды на этот ответ. Start-ноды остаются. Строка прозрачности: `Контекст: ноды проигнорированы, кроме start`.

## §5. Создание новой ноды

Перед созданием — обязательно:

1. **Проверить реестр** (`AGENTS.NODES.md`) и существующие ноды — нет ли уже ноды с тем же смыслом (по названию, триггерам, теме). Если есть — сообщить пользователю и предложить обновить существующую, а не создавать дубль.
2. **Уточнить у пользователя** если неясны: тема, назначение, триггеры, режим загрузки, имя файла. Не создавать наугад.
3. Если пользователь не знает как назвать ноду — агент **предлагает имя, краткое описание и структуру**, затем заполняет поля по согласованному смыслу.
4. После создания — обновить реестр.

## §6. Реестр

Все `*.node.md` регистрируются в `AGENTS.NODES.md`. Файлы `*.memory.md` и `*.metadata.md` в реестр не попадают.

**Протокол синхронизации реестра:**
1. Найти все `*.node.md` по всему хранилищу (от корня)
2. Сверить с реестром — добавить новые, обновить изменённые, удалить неактуальные
3. Проверить что все пути относительные от корня проекта
4. При смысловом изменении YAML-шапки ноды (приоритет, триггеры, статус и т.д.) — обновить соответствующую строку

Примеры нод — в `AGENTS.NODES.EXAMPLE.md` (по запросу пользователя).

## §7. Прозрачность в ответах

В начале каждого ответа — одна строка:

| Ситуация | Строка |
|----------|--------|
| Использована одна или несколько нод | `Контекст: использована нода <путь>` (пути через запятую) |
| Ноды не использовались | `Контекст: ответ без ноды` |
| Пользователь попросил «без нод» | `Контекст: ноды проигнорированы, кроме start` |

**Важно:** в строку контекста подставлять только реальные `*.node.md` — путь относительно корня проекта. Системные файлы `AGENTS.md`, `AGENTS.FULL.md`, `AGENTS.NODES.md`, `AGENTS.SETTINGS.md` нодами не являются — их указывать нельзя.

## §9. Метафора: клетки, узлы, синергия

Нода — как клетка организма: самодостаточный атом знания с понятной ролью. Цифровая экосистема строится из таких узлов — агент помнит и фиксирует происходящее в нужном файле, правит релевантные заметки, ориентируется где что лежит.

**Жёстких деревьев каталогов нет** — допустимы любые папки и файлы в vault. Важны не шаблоны папок, а договорённости и явные ноды.

Взаимодействие человека и ассистента — **синергия 1+1**: ясность агенту, опора и рост пользователю, без превращения памяти в кучу разрозненных дублей.

## §10. Где мы работаем

Vault Obsidian — workspace агента. Не выходить за его пределы без явного разрешения пользователя. Настройки окружения — в корневом `.env` (секреты не публиковать), шаблон — `.env.example`.

**Системная нода** — нода с `AWN-LOAD: start`. Она входит в базовый контекст полной сессии и не отключается режимом «без нод».

## §11. Команды

**Ноды и файлы:**
- Создать / изменить / удалить `*.node.md` → обновить реестр
- Создать / изменить / удалить `*.memory.md` → связать через `AWN-OWNER` (путь к ноде-владельцу)
- Создать / изменить / удалить sidecar `*.metadata.md` → рядом с бинарным файлом
- Обновить `.env` при изменении настроек окружения
- **Редактировать YAML-шапку** — добавлять, менять, удалять отдельные ключи в frontmatter `*.node.md`, `*.memory.md`, `*.metadata.md` без переписывания всего файла. Соблюдать порядок ключей (сначала `AWN-*`, затем пользовательские) и не ломать закрывающий `---`

**Автоматизация по расписанию (Cron):**
- **Обновить автоматизацию по Cron** — привести в соответствие раздел «Ноды с расписанием» в `AGENTS.NODES.md` (все ноды с `AWN-AUTOMATIZATION: true`, канон расписания — `AWN-CRON` в шапке) и внешнюю систему cron пользователя: для каждой такой ноды завести или обновить отдельную задачу по её `AWN-CRON`, удалить задачи для нод, которых больше нет в реестре; итог — актуальный список и в vault, и в планировщике

**Домены и категории:**
- Создать новый домен: завести подпапку `Domains/ИмяДомена/`, создать `Index.node.md` (NODE/INDEX) с `AWN-CATEGORY: <slug>`, создать папку `Assets/`, зарегистрировать в `Domains/Index.node.md` и `AGENTS.NODES.md`
- Добавить категорию: внести новую строку в таблицу `Domains/Categories.node.md`
- Удалить / архивировать домен: поставить `AWN-STATUS: archive` в `Index.node.md` домена, убрать из реестра `Domains/Index.node.md` и `AGENTS.NODES.md`
- Слаг категории (`AWN-CATEGORY`) — берётся из `Domains/Categories.node.md`; дефолт для системных нод — `system`, для остальных — `general`

**Мониторинг состояния:**
- Проверить актуальность реестра (все ли ноды на месте, нет ли дублей, верны ли пути)
- Проверить даты `AWN-UPDATED` и статусы `AWN-STATUS` — не залежались ли `draft`-ноды
- Прогон по `HEARTBEAT.md` при необходимости

**Развитие системы:**
- Фиксировать идеи улучшений в черновиках нод (`AWN-STATUS: draft`)
- Предлагать новые ноды если появляются повторяющиеся темы в запросах

**Согласованность:**
- Искать противоречия и пробелы между `AGENTS.md`, `AGENTS.FULL.md`, `AGENTS.NODES.md` и содержимым нод
- Предлагать архивацию (`AWN-STATUS: archive`) устаревших нод вместо удаления
