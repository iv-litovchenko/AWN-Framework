## Терминология

Протокол взаимодействия человека и ассистента (агент, машина) в структурированном пространстве памяти.  Фиксирует, как мы общаемся и где храним знания. **Scope:** Vault (Obsidian workspace) — совместное пространство агента и пользователя.

 **«Цифровое создание»**. Протокол превращает Obsidian из простого редактора текста в полноценную операционную систему для интеллекта которую ведут совместно человек и машина (ИИ-агент) - усиливая и дополняя друг друга 1+1>=2 - развиваясь вместе.

 Noda = любой объект системы в *.md формате перечисленные ниже в таблице. Все есть ноды - смысл память мирповедение логика.  Все остальные типы файлов сами по себе не являются нодами. 1 файл = 1 нода = 1 место. Каждая нода - это одна атомарная идея, запись в системе. По сути это путь и определение. Все является нодами в виде древовидной структуры.  Роль ноды определяется ее поведение и применение. Набор полей зависит от роли (типа) ноды. 

## Слои контекста
 
**Слой 1: Space Layer (уровень пространства). Определяет основной контекст границы (Boundary) и правила, по которым ИИ-агент понимает, где он находится: в рабочем проекте или в личном архиве. Контейнер, который задаёт контекст и группирует ноды. Это не просто папка — это _область мышления / namespace_. Пример space-financ/_index.node.md

**Слой 2: Part space Layer (расширение возможностей пространства части Space)**. Фундаментальный кирпич реальности внутри системы. Это самостоятельный объект.  Любой самостоятельный смысловой объект. Сущность. Явялется частью пространства. Пример user.node.md

**Слой 3: Resource и memory Layer (Уровень Ресурсов - данные Meta/Memory) Trace (След)** Meta / Storage / Memory (как это описано и хранится) память лог опыт заметки история событий

Это то самое «стабильное знание», которое ИИ может использовать как проверенный факт.


Память — это не склад, это именно след от произошедшего опыта. **Trace** идеально подчеркивает, что это хронологические отпечатки: логи чатов, сессий или дневниковые записи. Это не истина в последней инстанции, а «сырье» для последующего анализа и рефлексии ИИ-агентом.*

3.1 Meta Node (Метаданные)

**Что это:**   Описание  для файла (картинки, видео, аудио)
это _не контент_, а **описание контента**.

3.2 Memory Node (Память / журнал / заметки)
*.memory.md (Память / журнал / заметки)

**Что это:**  
Хронологическая или текстовая память, связанная с нодами или пространствами.

**Ключевая идея:**  
это не “знания”, а **следы опыта**.

- notes.md
- chat logs
- session logs
- daily logs

- список записей финансы
- список записей задачи

Ты правильно решил их игнорировать. В твоей системе они — **Reference (Справочник)**. Это "внешний шум" или "библиотека", которую ты не создавал как часть своей личности, а просто положил рядом для инфо-поддержки. Ссылки на них включаются в оглавление

### Резюме по архитектуре

Твоя схема решает главную проблему современных ИИ — **смешение контекстов**. Разделяя систему на эти три слоя, ты даешь агенту четкую инструкцию:

1. Сначала посмотри на **Anchor**, чтобы понять правила.
2. Найди нужные **Entities**, чтобы оперировать объектами.
3. Изучи **Traces**, чтобы понять историю вопроса и контекст прошлых взаимодействий.

Это превращает твой AWN-фреймворк из базы знаний в **динамическую среду обитания для ИИ**.

**Вопрос.** 

«Я так понимаю, что по большому счёту человек будет во всё это лазить только через ИИ-агента — просить его что-то сделать, поменять, открыть и так далее; сам я не буду искать и ходить по этим папкам и файлам, потому что там будет однотипная структура, которую очень хорошо понимает агент. Фактически это как операционка для агента?»

«Правильно ли я понимаю, что vault нужно проектировать как “операционную систему для ИИ-агента”, где человек работает в основном через агент, а структура файлов оптимизирована прежде всего для машинной предсказуемости, а не для ручной навигации?»

**Ответ.**

Да, ты очень точно сформулировал — это действительно похоже на операционную систему для агента.

И в такой модели приоритеты меняются:

- для человека важнее не ручной поиск по дереву, а понятные команды агенту;
- для агента важнее строгая, однотипная, предсказуемая структура;
- файловая система становится не “человеческим интерфейсом”, а “машинным API”.

Это абсолютно рабочий подход, если зафиксировать 3 вещи:

- стабильный контракт: одинаковые шаблоны, имена, правила ссылок;
- минимум исключений: только осознанные (`.DS_Store`, `.gitkeep`, возможно `sessions`);
- безопасные операции: агент делает массовые изменения только по подтверждённым сценариям (dry-run → apply).

То есть да: ты строишь не просто vault, а agent-native workspace.  
Человек работает на уровне намерений, агент — на уровне файлового протокола. Это как раз и есть “операционка для агента”.
