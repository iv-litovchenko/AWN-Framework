# Git Pull — получить изменения

> [!info] О файле
> Нода для **получения изменений** с remote. При **конфликте слияния** — не разрешать молча; остановиться и передать ситуацию пользователю.

## Назначение

Безопасно выполнить обновление из удалённого репозитория (`git pull` или `git pull --rebase` по правилам проекта) и корректно обработать **конфликты**.

## Алгоритм для агента

1. Выполнить **`git status`**. Если есть незакоммиченные изменения, **предупредить пользователя**: pull может привести к merge/rebase поверх грязного дерева. **Stash / коммит / отмена изменений** — только по **явной** инструкции пользователя, не скрывать риск.
2. Выполнить обновление:
   - **`git pull --rebase`**, если в этом репозитории так принято;
   - иначе **`git pull`** (merge).
3. **Если команда завершилась конфликтом** (в выводе есть `CONFLICT`, или `git status` показывает unmerged paths / «both modified»):
   - **немедленно остановиться**;
   - сообщить пользователю: краткий итог, вывод **`git status`**, список **конфликтных файлов**;
   - пояснить контекст: **merge** (после `git pull`) vs **rebase** (после `git pull --rebase`);
   - **не** править маркеры `<<<<<<<` / `=======` / `>>>>>>>` и **не** завершать merge/rebase коммитом **без явного указания пользователя**;
   - по запросу пользователя на откат незавершённого rebase: **`git rebase --abort`**; для merge — **`git merge --abort`** (только если пользователь просит).
4. Если конфликтов нет — кратко подтвердить успех; при необходимости показать **`git log -1 --oneline`**.

## Ограничения

- Не выполнять **`git push --force`** в рамках этой ноды.
- Не удалять или перезаписывать чужие изменения для «разруливания» конфликта без пользователя.

## Команды

```bash
git status
git pull --rebase    # или: git pull
```

При конфликте:

```bash
git status
```

По запросу пользователя (rebase):

```bash
git rebase --abort
```

По запросу пользователя (merge):

```bash
git merge --abort
```
