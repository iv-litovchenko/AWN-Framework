# Управление нодами

> [!info] Назначение
> Эта нода содержит правила и инструкции для работы с нодами AWN-Framework: создание, редактирование, регистрация в реестре и синхронизация.

## Создание новой ноды

### Шаг 1: Определить тип ноды

| Тип | Паттерн имени | Когда использовать |
|-----|---------------|-------------------|
| `NODE/SOLO` | `Имя.node.md` | Одиночная нода — правило, навык, контекст |
| `NODE/SPACE` | `_Index.node.md` | Точка сборки области, оркестратор папки |

### Шаг 2: Написать YAML-шапку

Сначала все поля `AWN-*`, затем пользовательские. Пример для `NODE/SOLO`:

```yaml
---
AWN-TYPE: NODE/SOLO
AWN-TITLE: Название ноды
AWN-DESC: "Зачем эта нода — 1-2 строки"
AWN-LOAD: start | on_demand
AWN-PRIORITY: 30
AWN-TRIGGERS: [слово1, слово2]
AWN-STATUS: active | disabled | draft | archive
AWN-CATEGORY: general
AWN-MEMORY: none | internal | external | hybrid
AWN-AUTOMATIZATION: false
AWN-VERSION: 1.0.0
AWN-CREATED: YYYY-MM-DD
AWN-UPDATED: YYYY-MM-DD
TAGS: [тег1, тег2]
---
```

### Шаг 3: Написать тело ноды

- Начать с блока `# Название`
- Добавить `> [!info] Назначение` с кратким описанием
- Описать правила, инструкции или контекст

### Шаг 4: Регистрация в реестре

После создания файла обязательно добавить запись в `Awn.Registry.md` в соответствующий раздел:

- `start` → Реестр нод «при старте»
- `on_demand` → Реестр нод «по необходимости»

Формат строки:
```
| active | priority | Путь к ноде | triggers | Описание |
```

## Редактирование ноды

1. Прочитать файл через `read_file`
2. Внести изменения (YAML или тело)
3. Обновить поле `AWN-UPDATED: YYYY-MM-DD`
4. Если изменились триггеры или приоритет — обновить реестр

## Синхронизация реестра

При массовых изменениях или если реестр кажется неактуальным:

1. Найти все `*.node.md` в хранилище
2. Сверить с `Awn.Registry.md`
3. Добавить отсутствующие, удалить неактуальные
4. Обновить метаданные изменившихся

## Принципы безопасной маршрутизации

> **Важно:** если не удаётся однозначно определить область для файла — **задать уточняющий вопрос** пользователю. Запрещено "угадывать" при неоднозначности (2+ равновероятных места).

## Проверка перед регистрацией

- [ ] Путь относителен от корня проекта (не начинается с `/Users/`)
- [ ] `AWN-STATUS: active` (иначе нода не загрузится)
- [ ] `AWN-TRIGGERS` заполнены осмысленно
- [ ] Файл сохранён в нужном месте






## §2. Ноды (`*.node.md`)

### Входной блок ноды

Сразу после закрывающего `---` шапки YAML — **сильная рекомендация** (не обязательное поле канона, но агент добавляет его осознанно при создании/обновлении нод):

```markdown
# Название области/ноды

> [!info] Об области/О ноде
> Зачем нода, что в ней, триггеры. Не путать с ...
```

Это помогает агенту и человеку сразу понять роль файла без чтения всего содержимого. Актуализировать при смысловых правках.

### Шаблон-заготовка: что может быть внутри ноды

Жёсткой типизации нет — автор выбирает нужные секции:

- **Назначение** — зачем нода нужна
- **Контекст** — где и в каких сценариях применяется
- **Факты/данные** — что хранить и обновлять
- **Правила/ограничения** — что можно и чего нельзя
- **Процесс/алгоритм** — как действовать пошагово
- **Триггеры** — при каких запросах подключать
- **Источники/ссылки** — на что опирается нода
- **Примеры** — краткие примеры применения
- **Журнал изменений** — опционально

По типу содержимого: `memory` → Факты + Обновлено; `process` → Шаги/Алгоритм; `rule` → Ограничения/Запреты; `reference` → Источники/Ссылки.

### Шаблоны скелетов

Готовые заготовки для каждого типа ноды — в папке [Templates/](Templates/):

- [Solo.template.md](Templates/Nodes/Solo.template.md) — одиночная нода (`NODE/SOLO`)
- [Index.template.md](Templates/Nodes/Index.template.md) — оркестратор папки (`NODE/INDEX`)
- [Area.template.md](Templates/Nodes/Area.template.md) — оболочка области (`NODE/AREA`)
- [Memory.template.md](Templates/Records/Memory.template.md) — файл внешней памяти (`*.memory.md`)
- [Metadata.template.md](Templates/Records/Metadata.template.md) — sidecar бинарного файла (`*.metadata.md`)
- [Note.template.md](Templates/Records/Note.template.md) — заметка домена (`Notes/`)

## §5. Создание новой ноды

Перед созданием — обязательно:

1. **Проверить реестр** (`AGENTS.NODES.md`) и существующие ноды — нет ли уже ноды с тем же смыслом (по названию, триггерам, теме). Если есть — сообщить пользователю и предложить обновить существующую, а не создавать дубль.
2. **Уточнить у пользователя** если неясны: тема, назначение, триггеры, режим загрузки, имя файла. Не создавать наугад.
3. Если пользователь не знает как назвать ноду — агент **предлагает имя, краткое описание и структуру**, затем заполняет поля по согласованному смыслу.
4. После создания — обновить реестр.

## §11. Команды

**Ноды и файлы:**
- Создать / изменить / удалить `*.node.md` → обновить реестр
- Создать / изменить / удалить `*.memory.md` → связать через `AWN-OWNER` (путь к ноде-владельцу)
- Создать / изменить / удалить sidecar `*.metadata.md` → рядом с бинарным файлом
- Обновить `.env` при изменении настроек окружения
- **Редактировать YAML-шапку** — добавлять, менять, удалять отдельные ключи в frontmatter `*.node.md`, `*.memory.md`, `*.metadata.md` без переписывания всего файла. Соблюдать порядок ключей (сначала `AWN-*`, затем пользовательские) и не ломать закрывающий `---`

**Автоматизация по расписанию (Cron):**
- **Обновить автоматизацию по Cron** — привести в соответствие раздел «Ноды с расписанием» в `AGENTS.NODES.md` (все ноды с `AWN-AUTOMATIZATION: true`, канон расписания — `AWN-CRON` в шапке) и **cron / планировщик окружения агента** (тот, где регистрируются периодические вызовы нод): для каждой такой ноды завести или обновить отдельную задачу по её `AWN-CRON`, удалить задачи для нод, которых больше нет в реестре; итог — актуальный список и в vault, и в планировщике

**Домены и категории:**
- Создать новый домен: завести подпапку `Domains/ИмяДомена/`, создать `Index.node.md` (NODE/INDEX) с `AWN-CATEGORY: <slug>`, создать папку `Assets/`, зарегистрировать в `Domains/Index.node.md` и `AGENTS.NODES.md`
- Добавить категорию: внести новую строку в таблицу `Domains/Categories.node.md`
- Удалить / архивировать домен: поставить `AWN-STATUS: archive` в `Index.node.md` домена, убрать из реестра `Domains/Index.node.md` и `AGENTS.NODES.md`
- Слаг категории (`AWN-CATEGORY`) — берётся из `Domains/Categories.node.md`; дефолт для системных нод — `system`, для остальных — `general`

**Мониторинг состояния:**
- Проверить актуальность реестра (все ли ноды на месте, нет ли дублей, верны ли пути)
- Проверить даты `AWN-UPDATED` и статусы `AWN-STATUS` — не залежались ли `draft`-ноды
- Прогон по `HEARTBEAT.md` при необходимости

**Развитие системы:**
- Фиксировать идеи улучшений в черновиках нод (`AWN-STATUS: draft`)
- Предлагать новые ноды если появляются повторяющиеся темы в запросах

**Согласованность:**
- Искать противоречия и пробелы между `AGENTS.md`, `AGENTS.FULL.md`, `AGENTS.NODES.md` и содержимым нод
- Предлагать архивацию (`AWN-STATUS: archive`) устаревших нод вместо удаления
