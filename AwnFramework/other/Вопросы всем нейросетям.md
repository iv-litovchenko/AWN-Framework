Это намного удобнее и надёжнее, чем чистые скиллы, потому что добавляет:

- Явную иерархию доверия (схема ноды vs произвольный контент).
    
- Триггерный механизм — агент не сканирует всё, а реагирует на слова/события.
    
- Версионирование и Git — легко откатывать изменения агента.
    
- Гибкость: ты можешь открыть проект в Cursor/VSCode и допрограммировать сложные ноды, а редактор через чат работает только с простыми.
    

Как ориентироваться в 1000+ md-файлах, не умирая от контекста?

Ты прав — пихать 1000 файлов в один промпт невозможно, модель "подохнет" (токены, потеря внимания, стоимость).

Как это решается в 2026 (и как можно улучшить AWN):

- Не грузить всё. Агент сначала работает только с реестром ([AGENTS.NODES.md](http://AGENTS.NODES.md)) — это маленький индекс.
    
- Потом использует RAG (Retrieval-Augmented Generation): векторная база (Weaviate, Qdrant, Pinecone, или даже pgvector в Postgres) на embeddings нод и memory.
    
    - Чанки нод (по 300–500 токенов + overlap).
        
    - Hybrid search: semantic (вектора) + keyword + metadata filters (по AWN-TYPE, tags, parentNote).
        
    - Агент сначала делает retrieval: "найди все ноды типа Филиал с триггером 'график'".
        
    - Только потом загружает полные релевантные ноды + их memory в контекст.
        
- Можно добавить summarization: для больших memory — держать короткий summary в YAML и полный текст отдельно.
    
- В AWN уже есть разделение (реестр vs on_demand) — это отличная база. Можно расширить: добавить автоматическую индексацию в векторную БД при создании/обновлении ноды (скрипт на Python или через MCP-tool).
    

Так агент ориентируется эффективно даже на десятках тысяч файлов.

## 3. Хранение заметок в Obsidian без папок

Да, можно отказаться от жёсткой структуры папок. Альтернативы:

- **Теги:** помечайте заметки тегами `#тег` (например, `#php`, `#проект_X`). Поиск по тегам заменяет навигацию по папкам.
    
- **Ссылки:** связывайте заметки ссылками `[[Название заметки]]`. Создавайте «хаб‑страницы» — центральные заметки, ссылающиеся на связанные темы.
    
- **Граф знаний:** используйте встроенный граф связей (Ctrl+G). Визуально видите, какие заметки связаны.
    
- **Поиск:** мощный поиск Obsidian (Ctrl+F, Ctrl+P) находит по содержимому, тегам, названиям.
    
- **Плагины:**
    
    - **Dataview:** запросы на языке SQL‑подобного синтаксиса (например, `list from #php` покажет все заметки с тегом `#php`).
        
    - **Advanced URI:** автоматизируйте создание и связывание заметок.
        
    - **QuickAdd:** быстрый ввод и категоризация.
        

**Принцип:** не «где это лежит?», а «как это найти?». Система ссылок и тегов делает расположение файла неактуальным.


## Основной принцип

Забудьте про иерархию «папка → подпапка → файл». Вместо этого:

- каждая заметка — **атом информации** (короткий фрагмент по одной теме);
    
- заметки связываются между собой **ссылками**;
    
- заметки группируются по темам с помощью **тегов**;
    
- поиск и навигация идут через **граф связей**, **поиск по содержимому** и **теги**.
    

## Пошаговая схема организации

**Шаг 1. Создайте единое хранилище (vault) без сложной структуры**

Просто создайте хранилище в Obsidian. Не создавайте заранее папки типа «Работа», «Учёба», «Личное» и т. д. Все заметки будут лежать в одной папке (или в нескольких базовых):

- `Inbox` — для быстрых записей, черновиков, идей «на потом»;
    
- `Archive` — для устаревших или завершённых заметок;
    
- основная папка (без названия) — для всех остальных заметок.
    

**Шаг 2. Пишите атомарные заметки**

Каждая заметка должна раскрывать **одну конкретную идею или факт**. Примеры:

- хорошо: «Принцип DRY в программировании», «Рецепт панкейков», «Встреча с Петей 15.05»;
    
- плохо: «Мои заметки по программированию и кулинарии», «Разные мысли».
    

**Шаг 3. Связывайте заметки ссылками**

В процессе написания заметки сразу добавляйте ссылки на связанные темы с помощью синтаксиса `[[Название заметки]]`.

Пример:

> В заметке «Принцип DRY» вы пишете: «Противоположность DRY — принцип WET ([[WET]])».  
> В заметке «Рецепт панкейков» добавляете: «Этот рецепт основан на базовом рецепте теста ([[Базовое тесто для выпечки]])».

**Шаг 4. Используйте теги для категоризации**

Добавляйте теги `#` в конце заметки или в начале, чтобы обозначить её тематику. Теги могут пересекаться.

Примеры тегов:

- `#программирование`, `#python`, `#рецепты`, `#выпечка`, `#встречи`, `#проект_Альфа`.
    

**Шаг 5. Наводите порядок по мере необходимости**

Не пытайтесь структурировать всё заранее. Делайте это, когда:

- в `Inbox` накопилось много черновиков — откройте их, допишите, свяжите ссылками с другими заметками, добавьте теги и перенесите в основное хранилище;
    
- вы заметили, что несколько заметок связаны общей темой — создайте «хаб‑заметку» с названием этой темы и добавьте в неё ссылки на все связанные заметки.
    

**Шаг 6. Пользуйтесь инструментами навигации**

- **Поиск (Ctrl+P / Ctrl+F):** находите заметки по названию или содержимому.
    
- **Граф знаний (Ctrl+G):** визуально видите, какие заметки связаны между собой. Ищите кластеры тем и недостающие связи.
    
- **Теги:** кликните на тег `#php` в любой заметке — Obsidian покажет все заметки с этим тегом.
    
- **Обратные ссылки (Backlinks):** в правой панели каждой заметки Obsidian показывает, какие другие заметки на неё ссылаются. Это помогает найти связанные идеи.

## Структура заметок с тегами (без жёстких папок)

**Базовый вариант (минимальная структура):**

- `Vault/` — корневая папка хранилища;
    
- `Inbox/` — быстрые записи и черновики;
    
- `Archive/` — завершённые/устаревшие заметки;
    
- `Templates/` — шаблоны заметок;
    
- `Assets/` или `Media/` — для всех вложений (картинки, аудио, PDF и т. д.);
    
- основная папка (в корне `Vault/`) — для всех рабочих заметок (без дополнительной иерархии).
    

**Как работают теги:**

Теги заменяют папки. Примеры:

- `#работа`, `#личное`, `#учёба` — глобальные категории;
    
- `#проект_Альфа`, `#клиент_Иванов` — проекты и клиенты;
    
- `#php`, `#дизайн`, `#маркетинг` — темы и навыки;
    
- `#idea`, `#todo`, `#done` — статусы;
    
- `#meeting_2024_05` — временные метки.

## Организация вложений (картинок и файлов)

**Вариант 1. Единая папка для всех вложений**

- папка: `Vault/Assets/` или `Vault/Media/`;
    
- внутри можно создать подпапки по типу:
    
    - `Images/`;
        
    - `Audio/`;
        
    - `PDFs/`;
        
    - `Others/`.
        

**Плюсы:** всё в одном месте, легко бэкапить.  
**Минусы:** может стать хаотично при тысячах файлов.

**Вариант 2. Вложенные папки рядом с заметками**

- для заметки `Vault/Мой проект.md` создаётся папка `Vault/Мой проект_assets/`;
    
- все вложения к заметке кладутся туда.
    

**Плюсы:** файлы привязаны к заметке, меньше путаницы.  
**Минусы:** разрозненность, сложнее бэкапить всё сразу.

**Вариант 3. Гибридный подход (рекомендуемый)**

- основная папка: `Vault/Assets/`;
    
- подпапки по **дате создания**: `Assets/2024/05/`, `Assets/2024/06/` и т. д.;
    
- или подпапки по **теме**: `Assets/Проекты/`, `Assets/Идеи/`, `Assets/Рецепты/`.
    

Это даёт баланс между порядком и гибкостью.

## Практические рекомендации по организации

**1. Именование файлов**

Чтобы избежать дубликатов и путаницы, используйте схему именования:

- по дате: `20240515_прототип_v2.jpg`;
    
- с префиксом темы: `проект_Альфа_диаграмма_архит.jpg`;
    
- уникальные ID (если используете плагины типа Obsidian UID).
    

**2. Ссылки на вложения в заметках**

Синтаксис в Markdown:

- картинка: `![Описание](путь/к/файлу.jpg)`;
    
- файл: `[Ссылка на PDF](путь/к/файлу.pdf)`.
    

Примеры:

- `![Схема базы данных](Assets/2024/05/db_schema_v1.jpg)`;
    
- `[ТЗ для дизайнера](Assets/Документы/ТЗ_дизайн_v3.pdf)`.
