**RAG расшифровывается:** Retrieval Augmented Generation — генерация с дополненным поиском.

---

**Популярные Vector DB для RAG**

|База|Особенность|
|---|---|
|**Qdrant**|быстрая, Docker, open source, хорошая фильтрация по metadata|
|**Chroma**|простая, встраивается в Python, для старта|
|**Pinecone**|облачная, managed, дорогая|
|**Weaviate**|своя схема данных, ближе к графу|
|**pgvector**|расширение PostgreSQL — если уже есть Postgres|
|**Milvus**|для больших масштабов|

Для твоего случая — **Qdrant** или **Chroma**. Qdrant лучше если есть Docker Compose.

---

**А про фундамент — ты прав**

В RAG-сообществе есть принципы, просто они менее формализованы чем наши ноды. Основные:

**Chunking strategy** — как резать текст на куски. Резать по абзацам, по предложениям, по фиксированному размеру. Плохой чанкинг = плохой поиск.

**Metadata filtering** — теги и поля которые сужают поиск до нужной области. Например `type: rule` или `domain: finance`.

**Embedding model** — какая модель превращает текст в вектор. От этого зависит качество поиска.

**Reranking** — после поиска топ-10 кусков — перепроверить их релевантность и оставить топ-3.

---

**Честная связь с нашими нодами**

Наши ноды фактически уже решают главную проблему RAG — **структуру и атомарность**. YAML-шапка это и есть metadata для фильтрации. `TYPE`, `TRIGGERS`, `STATUS` — готовые фильтры для Qdrant.

Большинство RAG-систем приходят к похожим идеям через боль и опыт. Мы заложили это с самого начала.

## Как устроена Vector DB

Обычная база данных хранит текст как текст. Поиск — по точному совпадению слов.

Vector DB хранит текст как **вектор** — массив из ~1500 чисел которые описывают **смысл** текста. Похожие по смыслу тексты имеют похожие векторы.

```
"расскажи анекдот"     → [0.2, 0.8, 0.1, ...]
"расскажи шутку"       → [0.21, 0.79, 0.11, ...]  ← близко
"финансовый отчёт"     → [0.9, 0.1, 0.7, ...]     ← далеко
```

Поиск работает не по словам а по **расстоянию между векторами**. Это и есть семантический поиск.

---

## Как данные хранятся

Каждая нода разбивается на чанки (куски) и для каждого создаётся запись:

```
{
  id: "randomAnecdote_chunk_1",
  vector: [0.2, 0.8, ...],        ← смысл текста
  metadata: {
    file: "Nodes/RandomAnecdote.node.md",
    TYPE: "Node/Domain",
    chunk: "Правило ответа: выбери короткий анекдот..."
  }
}
```

---

## Как это работает с твоими нодами

Сейчас:

```
Запрос → я читаю файл целиком → получаю весь контекст
```

С Vector DB:

```
Запрос → векторизуется → поиск похожих чанков → я получаю только релевантные куски
```

Твои `*.node.md` файлы идеально подходят для индексации — они атомарные, с чёткими YAML-полями. `TYPE`, `TRIGGERS`, `DESCRIPTION` — всё это metadata для фильтрации.

|Подход|Токены на запрос|
|---|---|

При росте системы экономия колоссальная.

## Чем лучше или хуже твоих нод

Не лучше и не хуже — это разные слои:

||Ноды (*.node.md)|Vector DB|
|---|---|---|
|Что хранит|Правила, договорённости|Смысл текста в числах|
|Кто читает|Человек + агент|Только агент|
|Поиск|По триггерам (слова)|По смыслу|
|Редактирование|Просто (Obsidian)|Нужна переиндексация|
|Инфраструктура|Не нужна|Docker + сервис|

Vector DB не заменяет ноды — она **индексирует** их содержимое для умного поиска.

## Как вписать в Docker Compose

```yaml
version: '3.8'
services:

  # Vector DB
  qdrant:
    image: qdrant/qdrant
    ports:
      - "6333:6333"
    volumes:
      - ./qdrant_data:/qdrant/storage

  # Индексатор нод
  indexer:
    build: ./indexer
    volumes:
      - /path/to/obsidian/vault:/vault:ro
    environment:
      - QDRANT_URL=http://qdrant:6333
      - WATCH_PATH=/vault
    depends_on:
      - qdrant

  # MCP сервер для Claude
  mcp-server:
    build: ./mcp
    ports:
      - "3000:3000"
    environment:
      - QDRANT_URL=http://qdrant:6333
    depends_on:
      - qdrant
```

**Что делает каждый контейнер:**

`qdrant` — сама база, хранит векторы нод.

`indexer` — следит за папкой Obsidian vault, при изменении `*.node.md` — переиндексирует файл. Пишется на Python за ~100 строк.

`mcp-server` — принимает запрос от Claude, векторизует его, ищет в Qdrant, возвращает релевантные куски нод.

---

## Полная схема с Obsidian

```
Obsidian vault (*.node.md)
        ↓ при изменении файла
    indexer (watchdog)
        ↓ разбивает на чанки, векторизует
    Qdrant (хранит векторы)
        ↑
    MCP server
        ↑
    Claude ← запрос пользователя
```

## Реальный совет

Сейчас это избыточно — у тебя одна нода. Но когда экосистема вырастет до 30+ нод — Qdrant + простой индексатор это ~200 строк кода и один `docker-compose up`. Стоит держать эту схему в голове при проектировании нод — именно поэтому атомарность (`APO`) важна: одна нода = один домен = хорошо индексируется.

**Как RAG хранит «правила дневника» без нод**

В классическом RAG нет понятия «нода» — есть просто куски текста с метаданными. Твоё правило о дневнике хранится так:

json

```json
{
  "text": "Дневник ведём каждый день в папке Diary/. 
           Формат файла: YYYY-MM-DD.md. 
           Начинаем с настроения, потом события, потом мысли.",
  "metadata": {
    "topic": "дневник",
    "type": "правило"
  }
}
```

Просто текст + теги. Никакого контракта, никакого YAML, никакой структуры. Каждая система придумывает своё.

---

**Как я запрашиваю RAG**

Я не запрашиваю сам. RAG — это внешний слой **до** меня:

```
Твой вопрос → MCP ищет в Qdrant → находит релевантные куски → 
добавляет их в мой контекст → я отвечаю
```

Я вижу уже готовый результат поиска. Как будто кто-то перед разговором положил мне нужные заметки на стол — я не знаю откуда они.

---

**И вот в чём проблема RAG без нод**

Через месяц у тебя в базе 500 кусков текста. Кто-то записал правило дневника три раза в разных формулировках — RAG находит все три, я получаю противоречия. Нет версий, нет статусов, нет `UPDATED`. Нет понятия «это правило устарело».

**Наши ноды решают именно это.** Нода — это структурированный контракт с версией, статусом, триггерами. RAG без нод это просто свалка текста с умным поиском.

---

**Идеальная связка**

```
*.node.md (структура, контракт, версия)
    ↓ индексируется
Vector DB (умный поиск по смыслу)
    ↓ релевантные куски
Claude (получает только нужное)
```

Ноды — это **источник истины**. RAG — это **способ доставки**. Одно без другого хуже.

---

**Что меняется в архитектуре**

Сейчас протокол такой:

```
Запрос → я читаю AGENTS.NODES.md → нахожу триггер → читаю файл ноды
```

С RAG:

```
Запрос → MCP ищет в Qdrant → подаёт мне куски нод → я отвечаю
```

Я больше **не читаю файлы сам**. Контекст приходит автоматически.

---

**Что меняется в AGENTS.md**

Раздел про загрузку нод переписывается. Вместо «читай по триггерам» → «доверяй контексту который пришёл от MCP»:

```
При работе с Vector DB:
- on_demand ноды не читаю вручную — 
  MCP уже подал релевантные куски в контекст
- start ноды — всё равно читаю при старте 
  (они всегда нужны независимо от запроса)
```

---

**Что меняется в AGENTS.NODES.md**

Таблица `on_demand` теряет смысл для агента — триггеры больше не нужны мне, они нужны вектору. Но остаётся нужной для **человека** — чтобы видеть какие ноды вообще существуют.

Таблица `start` остаётся полностью — эти ноды всегда загружаются явно.

---

**Что остаётся без изменений**

YAML-шапка каждой `*.node.md` — она становится **важнее**, не менее. `TYPE`, `TRIGGERS`, `STATUS` превращаются в metadata-фильтры в Qdrant. Структура нод идеально подходит для индексации.

---

**Итого: два режима работы**

Стоит зафиксировать в `AGENTS.md`:

|Режим|Как грузятся on_demand ноды|
|---|---|
|**Файловый** (сейчас)|читаю `AGENTS.NODES.md`, матчу триггеры, читаю файл|
|**RAG**|MCP подаёт куски автоматически, я не читаю файлы|

Система нод остаётся той же — меняется только **способ доставки** контекста ко мне.
