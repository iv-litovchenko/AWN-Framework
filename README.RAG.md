**RAG расшифровывается:** Retrieval Augmented Generation — генерация с дополненным поиском.

---

**Популярные Vector DB для RAG**

|База|Особенность|
|---|---|
|**Qdrant**|быстрая, Docker, open source, хорошая фильтрация по metadata|
|**Chroma**|простая, встраивается в Python, для старта|
|**Pinecone**|облачная, managed, дорогая|
|**Weaviate**|своя схема данных, ближе к графу|
|**pgvector**|расширение PostgreSQL — если уже есть Postgres|
|**Milvus**|для больших масштабов|

Для твоего случая — **Qdrant** или **Chroma**. Qdrant лучше если есть Docker Compose.

---

**А про фундамент — ты прав**

В RAG-сообществе есть принципы, просто они менее формализованы чем наши ноды. Основные:

**Chunking strategy** — как резать текст на куски. Резать по абзацам, по предложениям, по фиксированному размеру. Плохой чанкинг = плохой поиск.

**Metadata filtering** — теги и поля которые сужают поиск до нужной области. Например `type: rule` или `domain: finance`.

**Embedding model** — какая модель превращает текст в вектор. От этого зависит качество поиска.

**Reranking** — после поиска топ-10 кусков — перепроверить их релевантность и оставить топ-3.

**Честная связь с нашими нодами**

Наши ноды фактически уже решают главную проблему RAG — **структуру и атомарность**. YAML-шапка это и есть metadata для фильтрации. `TYPE`, `TRIGGERS`, `STATUS` — готовые фильтры для Qdrant.

Большинство RAG-систем приходят к похожим идеям через боль и опыт. Мы заложили это с самого начала.

## Как устроена Vector DB

Обычная база данных хранит текст как текст. Поиск — по точному совпадению слов.

Vector DB хранит текст как **вектор** — массив из ~1500 чисел которые описывают **смысл** текста. Похожие по смыслу тексты имеют похожие векторы.

```
"расскажи анекдот"     → [0.2, 0.8, 0.1, ...]
"расскажи шутку"       → [0.21, 0.79, 0.11, ...]  ← близко
"финансовый отчёт"     → [0.9, 0.1, 0.7, ...]     ← далеко
```

Поиск работает не по словам а по **расстоянию между векторами**. Это и есть семантический поиск.

---

## Как данные хранятся

Каждая нода разбивается на чанки (куски) и для каждого создаётся запись:

```
{
  id: "randomAnecdote_chunk_1",
  vector: [0.2, 0.8, ...],        ← смысл текста
  metadata: {
    file: "Nodes/RandomAnecdote.node.md",
    TYPE: "Node/Domain",
    chunk: "Правило ответа: выбери короткий анекдот..."
  }
}
```

## Как это работает с твоими нодами

Сейчас:

```
Запрос → я читаю файл целиком → получаю весь контекст
```

С Vector DB:

```
Запрос → векторизуется → поиск похожих чанков → я получаю только релевантные куски
```

Твои `*.node.md` файлы идеально подходят для индексации — они атомарные, с чёткими YAML-полями. `TYPE`, `TRIGGERS`, `DESCRIPTION` — всё это metadata для фильтрации.

## Чем лучше или хуже твоих нод

Не лучше и не хуже — это разные слои:

||Ноды (*.node.md)|Vector DB|
|---|---|---|
|Что хранит|Правила, договорённости|Смысл текста в числах|
|Кто читает|Человек + агент|Только агент|
|Поиск|По триггерам (слова)|По смыслу|
|Редактирование|Просто (Obsidian)|Нужна переиндексация|
|Инфраструктура|Не нужна|Docker + сервис|

Vector DB не заменяет ноды — она **индексирует** их содержимое для умного поиска.

## Как вписать в Docker Compose

```yaml
version: '3.8'
services:

  # Vector DB
  qdrant:
    image: qdrant/qdrant
    ports:
      - "6333:6333"
    volumes:
      - ./qdrant_data:/qdrant/storage

  # Индексатор нод
  indexer:
    build: ./indexer
    volumes:
      - /path/to/obsidian/vault:/vault:ro
    environment:
      - QDRANT_URL=http://qdrant:6333
      - WATCH_PATH=/vault
    depends_on:
      - qdrant

  # MCP сервер для Claude
  mcp-server:
    build: ./mcp
    ports:
      - "3000:3000"
    environment:
      - QDRANT_URL=http://qdrant:6333
    depends_on:
      - qdrant
```

**Что делает каждый контейнер:**

`qdrant` — сама база, хранит векторы нод.

`indexer` — следит за папкой Obsidian vault, при изменении `*.node.md` — переиндексирует файл. Пишется на Python за ~100 строк.

`mcp-server` — принимает запрос от Claude, векторизует его, ищет в Qdrant, возвращает релевантные куски нод.

## Что такое MCP и как создать

**MCP** (Model Context Protocol) — протокол Anthropic для подключения внешних инструментов к Claude. По сути это сервер который регистрирует набор инструментов — Claude может их вызывать как функции.

Структура MCP сервера:

```python
from mcp import MCPServer, tool

server = MCPServer("awn-nodes")

@tool
def search_nodes(query: str) -> str:
    """Найти релевантные ноды по запросу"""
    # поиск в Qdrant
    return relevant_chunks

@tool  
def read_node(path: str) -> str:
    """Прочитать конкретную ноду"""
    return open(path).read()

@tool
def update_node(path: str, content: str) -> str:
    """Обновить ноду"""
    open(path, 'w').write(content)
    return "updated"

server.run()
```

Правила MCP:

- Каждый инструмент имеет имя, описание, параметры
- Claude сам решает когда вызвать инструмент по описанию
- Результат возвращается в контекст Claude
- Транспорт: stdio или HTTP

Подключение в `claude_desktop_config.json`:

```json
{
  "mcpServers": {
    "awn": {
      "command": "python",
      "args": ["awn_mcp.py"]
    }
  }
}
```

## ACP и другие протоколы агентного общения

**ACP** (Agent Communication Protocol) — более новый стандарт, позволяет агентам общаться между собой напрямую, не только через человека.

Ландшафт протоколов:

|Протокол|Что делает|Кто сделал|
|---|---|---|
|**MCP**|Агент ↔ Инструменты/данные|Anthropic|
|**ACP**|Агент ↔ Агент|IBM/Linux Foundation|
|**A2A**|Агент ↔ Агент|Google|
|**AutoGen**|Мультиагентные сценарии|Microsoft|
|**LangGraph**|Граф агентов с состоянием|LangChain|

**Как агенты общаются по ACP:**

```
Пользователь → Claude (оркестратор)
                    ↓
              Claude вызывает Gemini как агента
                    ↓
              Gemini возвращает результат
                    ↓
              Claude собирает финальный ответ
```

Каждый агент в сети — это сервис с API. Оркестратор (обычно основная модель) решает кого вызвать и как собрать результат.

**В контексте твоей системы** это могло бы выглядеть так: Claude работает с тобой, но для конкретных задач (код → Codex, изображения → другая модель) делегирует подзадачи другим агентам — все они знают твои ноды.

**Redis**

Это key-value хранилище в памяти (RAM). Очень быстрое. Для агентов используется как:

- **Кэш сессии** — помнить что происходило в текущем разговоре без перечитывания файлов
- **Очередь задач** — если агент должен что-то сделать позже
- **Счётчики** — сколько раз нода использовалась, когда последний раз

Для твоей системы конкретно: Redis мог бы хранить «какие ноды уже загружены в этой сессии» — не читать файл повторно, а брать из кэша. Но при 200к контексте это не критично.

---

**Mem0**

Специализированная память для агентов. Делает три вещи:

```
Разговор → Mem0 извлекает факты → сохраняет структурированно

"Я предпочитаю короткие ответы" → { user_preference: "short_answers" }
"Мой бюджет на кофе — 5000р/мес" → { finance: { coffee: 5000 } }
```

По сути это **автоматическое создание нод** из разговора. Mem0 сам решает что важно запомнить.

Проблема — он решает сам, без твоего контроля. Наши ноды ты создаёшь осознанно.

**Как люди структурируют память агентов в БД**

Обычно четыре типа памяти — каждый своя таблица:

**1. Факты о пользователе (семантическая память)**

```sql
CREATE TABLE facts (
  id UUID PRIMARY KEY,
  subject VARCHAR,    -- "пользователь", "проект AWN"
  predicate VARCHAR,  -- "предпочитает", "работает над"
  object TEXT,        -- "короткие ответы", "Obsidian"
  confidence FLOAT,   -- 0.0–1.0
  source VARCHAR,     -- из какого разговора
  created_at TIMESTAMP,
  expires_at TIMESTAMP  -- память может устаревать
);
```

**2. История разговоров (эпизодическая память)**

```sql
CREATE TABLE episodes (
  id UUID PRIMARY KEY,
  session_id VARCHAR,
  role ENUM('user', 'assistant'),
  content TEXT,
  timestamp TIMESTAMP,
  summary TEXT  -- сжатое резюме для экономии токенов
);
```

**3. Правила и договорённости (процедурная память)**

```sql
CREATE TABLE procedures (
  id UUID PRIMARY KEY,
  domain VARCHAR,      -- "финансы", "дневник"
  name VARCHAR,        -- "как вести записи"
  content TEXT,        -- само правило
  version VARCHAR,     -- 1.0.0
  status ENUM('draft','active','archive'),
  updated_at TIMESTAMP
);
```

**4. Контент-записи (фактические данные)**

```sql
CREATE TABLE content_records (
  id UUID PRIMARY KEY,
  type VARCHAR,        -- "финансы", "дневник"
  date DATE,
  data JSON,           -- сама запись
  created_at TIMESTAMP
);
```

---

**Ключевое наблюдение**

Смотри — `procedures` это и есть наши `*.node.md` в SQL виде. `content_records` это наши `*.content.md`. Мы уже придумали правильную структуру — просто в Markdown формате.

**Как решить баланс между Obsidian и MySQL**

Три подхода которые используют на практике:

**Подход А — Obsidian как источник истины**

```
*.node.md → синхронизатор → MySQL (только для запросов)
```

Ты редактируешь в Obsidian, скрипт синхронизирует в MySQL для аналитики.

**Подход Б — MySQL как источник истины**

```
MySQL → генератор → *.md файлы (только для чтения человеком)
```

Агент пишет в MySQL, скрипт генерирует красивые md-файлы.

**Подход В — разделить по типу данных**

```
Правила/ноды → Obsidian (ты их редактируешь)
Контент/записи → MySQL (агент их создаёт)
```

Для твоей системы **Подход В** самый честный: ноды живут в Obsidian — ты их контролируешь. Финансовые записи, задачи, дневник — идут в MySQL, агент их создаёт и обновляет. Отдельный скрипт может генерировать Obsidian-представление для чтения.

**Таблица 1 — `facts` (семантическая память)**

Факты о пользователе которые агент узнал и должен помнить всегда.

```sql
CREATE TABLE facts (
  id          UUID PRIMARY KEY,
  subject     VARCHAR,   -- о чём факт: "пользователь", "проект_AWN"
  predicate   VARCHAR,   -- тип связи: "предпочитает", "работает_над", "имеет"
  object      TEXT,      -- само значение: "короткие ответы без списков"
  confidence  FLOAT,     -- 0.0–1.0: насколько уверены
  source      VARCHAR,   -- откуда узнали: "сессия_2026-04-16"
  created_at  TIMESTAMP,
  expires_at  TIMESTAMP  -- NULL = бессрочно
);
```

Примеры строк:

```
subject="пользователь"  predicate="предпочитает"  object="отвечай без буллитов"
subject="пользователь"  predicate="язык"          object="русский"
subject="AWN"           predicate="статус"         object="в разработке"
```

Это и есть «кто я» нода в SQL виде.

**Таблица 2 — `episodes` (эпизодическая память)**

Что происходило в прошлых сессиях. Не весь текст — только резюме.

```sql
CREATE TABLE episodes (
  id          UUID PRIMARY KEY,
  session_id  VARCHAR,
```

**Таблица 3 — `procedures` (процедурная память)**

Как делать конкретные вещи. Это прямой аналог наших `*.node.md`.

```sql
CREATE TABLE procedures (
  id          UUID PRIMARY KEY,
  domain      VARCHAR,    -- "финансы", "дневник", "задачи"
  name        VARCHAR,    -- "как вести ежедневные записи"
  content     TEXT,       -- само правило/инструкция
  triggers    JSON,       -- ["финансы", "расход", "доход"]
  require     ENUM('start','on_demand'),
  version     VARCHAR,    -- "1.0.0"
  status      ENUM('draft','active','archive'),
  priority    INT,        -- 1–100
  created_at  TIMESTAMP,
  updated_at  TIMESTAMP
);
```

Видишь? Это точная копия YAML-шапки нашей ноды в SQL. Разница только в том что тут можно делать запросы:

```sql
-- Все активные ноды которые грузятся при старте
SELECT * FROM procedures 
WHERE status='active' AND require='start'
ORDER BY priority ASC;

-- Найти ноды по триггеру
SELECT * FROM procedures
WHERE JSON_CONTAINS(triggers, '"финансы"');
```

**Таблица 4 — `content_records` (фактические данные)**

Сами записи — финансы, дневник, задачи. Аналог наших `*.content.md`.

```sql
CREATE TABLE content_records (
  id          UUID PRIMARY KEY,
  type        VARCHAR,    -- "финансы", "дневник", "задача"
  date        DATE,
  data        JSON,       -- вся запись в свободном формате
  tags        JSON,       -- ["расход", "еда"]
  status      ENUM('draft','active','archive'),
  created_at  TIMESTAMP,
  updated_at  TIMESTAMP
);
```

Примеры строк:

```json

data={"сумма": -320, "категория": "еда", "комментарий": "обед"}

type="дневник"  date="2026-04-16"
data={"настроение": "хорошее", "события": "обсудили RAG с агентом"}
```

Получается если я подключу RAG - то ты напорядок уменьшишь кол-во потребялемых токенов в этом фреймворке-подходе?

Да, но с важной оговоркой.

**Где RAG реально экономит токены:**

При большом количестве нод — да, сильно. Вместо того чтобы читать все ноды при старте или по триггеру, RAG подаёт только релевантные куски.

```
Сейчас (10 нод, все при старте):  ~15 000 токенов
С RAG (10 нод, топ-3 чанка):       ~500–800 токенов
```

**Где RAG НЕ экономит:**

Сама переписка — основной потребитель токенов. Наш сегодняшний разговор это ~50 000 токенов. RAG на это не влияет никак.

---

**Честная картина:**

|Компонент|Без RAG|С RAG|
|---|---|---|
|Ноды при старте|~4 000|~300|
|On_demand ноды|~500–1000 за штуку|~150–300|
|Переписка|~50 000|~50 000|
|**Итого**|**~55 000**|**~51 000**|

При текущем размере системы — экономия ~5–10%. Незначительно.

При 50+ нодах — картина меняется. Без RAG старт мог бы стоить 30 000–50 000 токенов только на ноды. С RAG — всегда ~300–500 независимо от количества нод.

**Вывод:** RAG нужен не для экономии токенов сейчас, а для **масштабируемости** — чтобы система не деградировала при росте.
